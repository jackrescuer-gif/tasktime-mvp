# История архитектурной документации

**Цель:** Пояснить, какие документы вынесены в архив и почему для MVP оставлен минимальный набор.

**Дата:** март 2025

---

## 1. Итоговый набор документов для MVP

Для реализации и поддержки MVP актуальны только четыре документа (в `docs/ENG/architecture/` и `docs/RU/architecture/`):

| Документ | Назначение |
|----------|------------|
| **SYSTEM_ARCHITECTURE.md** | Единая точка входа: диаграмма системы, целевая доменная модель, модули, дорожная карта и указатель на остальные документы. |
| **MVP_DOMAIN_MODEL.md** | Целевые сущности и связи (ER, атрибуты). Всё, что нужно для реализации схемы БД и API. |
| **REPO_RESTRUCTURE_PLAN.md** | Целевая структура репозитория и правила переноса кода в модули. |
| **ISSUE_ENGINE.md** | Жизненный цикл задачи: статусы, доска, спринт, история. Критично для логики задач и канбана. |

Остальная архитектурная документация перенесена в **docs/archive/** (подпапки ENG и RU), чтобы не удалять её навсегда, но не отвлекать от MVP.

---

## 2. Что перенесено в архив и почему

### 2.1. CURRENT_ARCHITECTURE.md

**Содержало:** Снимок текущего стека (Node/Express, PostgreSQL, фронт без сборки), структура server.js, фронта и БД, риски.

**Почему не нужно для MVP:** Описание «как есть» дублируется в SYSTEM_ARCHITECTURE.md (раздел про текущее состояние) и в REPO_RESTRUCTURE_PLAN.md (что откуда переносить). Для MVP достаточно целевой картины и плана реструктуризации; детальный «as-is» полезен только на время миграции и сохранён в архиве.

### 2.2. DOMAIN_MODEL.md

**Содержало:** Анализ текущей схемы БД (tasks, task_items, product_teams и т.д.), диаграмма сущностей, пробелы относительно целевой модели.

**Почему не нужно для MVP:** Целевая модель полностью описана в MVP_DOMAIN_MODEL.md. Текущая схема нужна лишь для миграции (tasks → issues); ключевые решения по миграции отражены в SYSTEM_ARCHITECTURE.md (фазы) и в MIGRATION_PLAN.md (в архиве). Для ежедневной разработки MVP достаточно MVP_DOMAIN_MODEL.md.

### 2.3. ARCHITECTURE_PROPOSAL.md

**Содержало:** Предложение модульного монолита: принципы, список модулей, слои (api → service → repository), правила зависимостей.

**Почему не нужно для MVP:** Итог принят и включён в SYSTEM_ARCHITECTURE.md (раздел модулей и принципы) и в REPO_RESTRUCTURE_PLAN.md (структура папок и границы модулей). Отдельный «предложенческий» документ создаёт дублирование; для реализации достаточно двух указанных.

### 2.4. MIGRATION_PLAN.md

**Содержало:** Пофазный план миграции (Phase 0–4), шаги, проверки, откат.

**Почему не нужно для MVP:** Краткая дорожная карта есть в SYSTEM_ARCHITECTURE.md (таблица фаз и рисков). Детальный план по шагам нужен при выполнении миграции; для понимания архитектуры и приоритетов достаточно сводки. Документ сохранён в архиве для использования при планировании фаз.

### 2.5. PERFORMANCE_PLAN.md

**Содержало:** Стратегии по большим доскам, пагинации, индексам, кэшированию, виртуализации списков.

**Почему не нужно для MVP:** Оптимизация производительности — следующий этап после стабилизации функциональности. Для запуска MVP достаточно базовых ограничений (limit/offset), которые можно описать в API и в ISSUE_ENGINE. Детальный план производительности не обязателен на старте и сохранён в архиве.

### 2.6. KANBAN_ARCHITECTURE.md

**Содержало:** Устройство канбана: колонки, drag-and-drop, оптимистичные обновления, порядок карточек, API перемещения.

**Почему не нужно для MVP:** Жизненный цикл задачи и связь с доской/статусами описаны в ISSUE_ENGINE.md. Конкретика по drag-and-drop и UX канбана может быть вынесена в задачи/спеки по фронту; для единой точки входа по «движку» задач достаточно ISSUE_ENGINE. Документ в архиве — для детальной реализации канбана.

### 2.7. AI_AGENTS.md

**Содержало:** Описание агентов (Architect, Backend, Frontend, DB, Refactor, QA), их роли и подсказки для вызова.

**Почему не нужно для MVP:** Настройка агентов и промптов живёт в .cursor/ (rules, skills, AGENTS). Архитектурные документы должны описывать систему, а не инструменты разработки. Ссылки на архитектуру из промптов можно давать на SYSTEM_ARCHITECTURE.md и три остальных документа. Файл в архиве — для справки по ролям агентов.

### 2.8. CURSOR_RULES.md

**Содержало:** Правила для AI: именование, структура папок, API, БД, безопасность.

**Почему не нужно для MVP:** Такие правила должны находиться в .cursor/rules/ или в корневом cursorrules и применяться редактором/агентами. Дублирование в docs/architecture размывает единственный источник правды. Документ в архиве — для переноса правил в .cursor при необходимости.

## 3. Какие решения сохранены

- **Целевая архитектура:** модульный монолит, один Node.js (Express), одна БД PostgreSQL; Nginx как обратный прокси. Описано в SYSTEM_ARCHITECTURE.md.
- **Целевая доменная модель:** Organization (опционально), User, Team, Project, Issue, Comment, Sprint, Board, BoardColumn, Label, TimeLog, AuditLog. Детально в MVP_DOMAIN_MODEL.md.
- **Модули и границы:** auth, users, projects, issues, comments, boards, sprints, time, teams, audit, admin; слой api → service → repository без кросс-импорта репозиториев. В SYSTEM_ARCHITECTURE.md и REPO_RESTRUCTURE_PLAN.md.
- **Дорожная карта:** фазы 0–4 (подготовка → выделение модулей → новый домен → объединение tasks в issues → фронт и опционально организации). В SYSTEM_ARCHITECTURE.md.
- **Жизненный цикл задач:** статусы, доска, спринт, история. В ISSUE_ENGINE.md.

Никакие из перечисленных решений не отменены; избыточные и вспомогательные документы вынесены в архив для упрощения навигации и работы с документацией в рамках MVP.
