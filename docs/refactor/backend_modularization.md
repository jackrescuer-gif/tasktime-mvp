# Рефакторинг бэкенда: модульный монолит

## Цель

Преобразование бэкенда из одного крупного `server.js` в модульную структуру без изменения поведения приложения и схемы БД.

---

## Созданные модули

### 1. **auth**
- **api.js** — маршруты: регистрация, вход, выход, текущий пользователь, имперсонация.
- **service.js** — хеширование пароля, проверка блокировки, выдача JWT, вызов audit.
- **repository.js** — вставка пользователя, поиск по email/id, проверка is_blocked.

### 2. **users**
- **api.js** — один маршрут: список пользователей (для выбора исполнителя).
- **service.js** — делегирование в репозиторий.
- **repository.js** — выборка id, email, name, role по имени.

### 3. **projects**
- **api.js** — список проектов, создание, получение по id (с эпиками).
- **service.js** — проверка прав (только admin/manager на создание), вызов audit при создании, сборка ответа с эпиками.
- **repository.js** — список с агрегатами (epics_count, stories_count, done_count), создание, выборка по id, эпики по project_id.

### 4. **issues**
- **api.js** — CRUD по задачам (tasks), связки task–task_item (task-links), CRUD по иерархическим элементам (task-items), смена статуса.
- **service.js** — RBAC (canReadTask, canUpdateTask, canDeleteTask, canReadAll, isReadOnly), валидация иерархии (epic→story→subtask), audit, вызов webhook при создании задачи.
- **repository.js** — все SQL по tasks, task_items, task_item_links; построение дерева (buildTree); fallback-запросы при отсутствии колонки project_id.

### 5. **time**
- **api.js** — старт/стоп таймера по задаче, список логов по задаче, список своих логов (с опциональным фильтром по task_id).
- **service.js** — проверка доступа к задаче (canReadTask), проверка отсутствия уже запущенного таймера, вызов audit.
- **repository.js** — выборка задачи для RBAC, поиск открытого таймера, вставка/обновление time_logs, списки по task_id и user_id.

### 6. **admin**
- **api.js** — системная статистика, список пользователей, обновление пользователя (роль/блокировка), лог активности, история деплоев, дашборды CIO и Main, бизнес-функции и продуктовые команды (CRUD и участники).
- **service.js** — форматирование uptime, разбор лога деплоев, правила обновления пользователя (не себя, только super-admin для admin/super-admin), сборка ответов дашбордов.
- **repository.js** — все SQL для admin/dashboard/business-functions/product-teams; чтение файла лога деплоев.

---

## Общие модули (shared)

- **auth.js** — middleware аутентификации (JWT из заголовка или куки), adminMiddleware, функции проверки прав (canReadTask, canUpdateTask, canDeleteTask, isAdminOrManager, canReadAll, isReadOnly), константы куки.
- **audit.js** — запись в audit_log (user_id, action, entity_type, entity_id, level, details, ip, user_agent); перенесён из корня backend, все вызовы идут из модулей.
- **errors.js** — вспомогательная отправка JSON-ошибки (sendError).

---

## Конфигурация и точка входа

- **config.js** — PORT, NODE_ENV, JWT_SECRET, JWT_EXPIRES_IN, COOKIE_NAME, COOKIE_OPTS, PIXEL_OFFICE_WEBHOOK_URL, DEPLOY_LOG_PATH (без логики).
- **db.js** — без изменений: пул PostgreSQL, query, getClient.
- **server.js** — только: инициализация Express, подключение middleware (json, cookieParser), подключение роутеров модулей (auth, users, projects, time, issues, admin), маршруты health, /admin, /, /app, статика, запуск сервера. Никакой бизнес-логики и SQL.

---

## Как изменился server.js концептуально

Раньше в одном файле были: конфиг, middleware, все маршруты, вся бизнес-логика и все SQL-запросы. Теперь server.js только собирает приложение: подключает общие middleware и монтирует по одному роутеру на домен (auth, users, projects, issues, time, admin). Вся логика и работа с БД вынесены в модули. Порядок монтирования сохранён так, чтобы более специфичные пути (например, `/api/tasks/:id/time/start`) обрабатывались нужным модулем (time).

---

## Разделение ответственности

- **api.js** — только определение маршрутов и обработка запроса/ответа: извлечение параметров из req, вызов service, отправка кодов и JSON. Никакой бизнес-логики и SQL.
- **service.js** — бизнес-правила: права доступа, валидация, оркестрация репозитория и shared (audit, webhook). Не содержит SQL и деталей HTTP.
- **repository.js** — единственное место с SQL и обращением к БД (включая чтение файла лога деплоев в admin). Никакой проверки прав и бизнес-логики.

В результате запрос обрабатывается цепочкой: api → service → repository (и при необходимости shared/auth, shared/audit).

---

## Зачем такая архитектура удобна для ИИ-инструментов и разработчиков

1. **Меньший контекст** — ИИ или разработчик может открыть один модуль (например, только issues) и видеть только релевантные маршруты, логику и запросы, без просмотра всего server.js.
2. **Чёткие границы** — по имени файла понятно, искать ли маршрут (api), правило (service) или запрос к БД (repository). Поиск по коду и рефакторинг становятся предсказуемыми.
3. **Единое место для SQL** — все запросы в repository.js соответствующего модуля; проще проверять безопасность (параметризация), индексы и миграции.
4. **Повторное использование** — сервисы можно вызывать из других сервисов или из тестов без поднятия HTTP; репозитории изолированы по домену.
5. **Модульный монолит** — без микросервисов и сетевых границ; при необходимости любой модуль позже можно выделить в отдельный сервис, так как зависимости уже ограничены слоями api → service → repository.

Схема БД в этом рефакторинге не менялась; изменена только структура кода бэкенда.
