## Тест-план: область проекта «MVP замены Jira на вайб-коде»

### 1. Unit / API-тесты

#### 1.1. Модель и сидирование проекта

- [ ] При выполнении `seed.js` в таблице `projects` создаётся запись:
  - `name = 'MVP замены Jira на вайб-коде'`
  - `project_type = 'real'`
- [ ] Повторный запуск `seed.js` не создаёт дублирующие записи (idempotent insert c `ON CONFLICT`).

#### 1.2. `GET /api/projects`

- [ ] Эндпоинт возвращает список проектов со всеми полями, включая `project_type`.
- [ ] Для существующих демо-проектов `project_type = 'demo'`.
- [ ] Для проекта «MVP замены Jira…» `project_type = 'real'`.
- [ ] Поля агрегатов по иерархическим задачам (`epics_count`, `stories_count`, `done_count`) заполняются без ошибок (запрос не падает при отсутствии связанных `task_items`).

#### 1.3. `GET /api/projects/:id`

- [ ] Для id проекта «MVP замены Jira…» возвращается объект с теми же полями, что и в списке, плюс массив `epics`.
- [ ] При запросе несуществующего id возвращается 404.

#### 1.4. `GET /api/task-items?context_type=project&context_id=<id>&level=story`

- [ ] Для проекта «MVP замены Jira…» возвращается массив сториз (уровень `story`), привязанных к этому проекту.
- [ ] При отсутствии сториз запрос возвращает пустой массив, без ошибок.

#### 1.5. `PATCH /api/task-items/:id/status`

- [ ] Разрешены статусы `open`, `in_progress`, `in_review`, `done`, `cancelled`.
- [ ] Попытка установить любое другое значение даёт 400 `Invalid status`.
- [ ] Пользователь без прав (не admin/manager и не автор/исполнитель) получает 403.
- [ ] Изменение статуса фиксируется в `audit_log` с экшеном `task_item.status_changed`.

### 2. UI-сценарии: список проектов и разделение real/demo

#### 2.1. Страница «Проекты»

Ожидаемое поведение:

- [ ] В верхней части страницы есть блок «Живые продуктовые проекты» (Real project).
- [ ] В этом блоке отображается как минимум один проект с бейджем **MVP Jira** и именем «MVP замены Jira на вайб-коде».
- [ ] Ниже расположен блок «Демонстрационные проекты» с остальными проектами.
- [ ] Если демо-проекты временно отсутствуют, блок «Демонстрационные проекты» показывается с текстом «Нет демо-проектов» и без ошибок в консоли.

#### 2.2. Визуальное отличие реал/демо

- [ ] Карточка реального проекта визуально выделена (рамка/тень/бейдж), заметно отличаясь от демо.
- [ ] Карточки демо-проектов имеют нейтральный стиль и бейдж «Demo» в заголовке раздела.

### 3. UI-сценарии: Kanban-доска проекта «MVP замены Jira…»

#### 3.1. Отображение колонок

- [ ] При открытии карточки проекта «MVP замены Jira…» над деревом задач виден блок «Kanban — сториз проекта».
- [ ] Отображаются три колонки: **Backlog**, **In progress**, **Done**.
- [ ] В колонках отображаются сториз этого проекта:
  - `status = 'open'` → Backlog;
  - `status = 'in_progress'` и `status = 'in_review'` → In progress;
  - `status = 'done'` → Done.
- [ ] Если в колонке пока нет задач, отображается аккуратный текст «Нет задач», без визуальных артефактов.

#### 3.2. Смена статуса из Kanban

Для любой сториз проекта «MVP замены Jira…»:

- [ ] Кнопка **Backlog** переводит задачу в колонку Backlog (статус `open`).
- [ ] Кнопка **In progress** переводит задачу в колонку In progress (статус `in_progress`).
- [ ] Кнопка **Done** переводит задачу в колонку Done (статус `done`).
- [ ] После клика карточка визуально «переезжает» в нужную колонку без перезагрузки всей страницы.

Граничные случаи:

- [ ] При временной потере сети/сбое бэкенда кнопка показывает ошибку (alert или сообщение), при этом страница не «ломается» и остаётся в рабочем состоянии.
- [ ] При повторном открытии карточки проекта статусы корректно перечитываются с сервера.

### 4. Роли и доступ

- [ ] Роль `viewer` видит страницу «Проекты» и Kanban‑доску, но не может менять статусы (кнопки либо скрыты, либо неактивны).
- [ ] Роли `admin` и `manager` могут изменять статусы сториз через Kanban.
- [ ] Попытка изменить статус без авторизации или с истёкшей сессией приводит к редиректу на страницу логина.

### 5. Регрессия демо-проектов

- [ ] Карточки демо-проектов по‑прежнему открываются и показывают дерево задач.
- [ ] Отображение дерева задач (эпики/сториз/сабтаски) не изменилось и работает как раньше.
- [ ] Никакие существующие сценарии dashboard / quick tasks / teams не ломаются из‑за появления `project_type`.
