# Инструкция администратора TaskTime

Установка, настройка, учётные записи, резервное копирование, логи и аудит, рекомендации по безопасности.

---

## Установка и переменные окружения

Основные шаги развёртывания описаны в [DEPLOY.md](../DEPLOY.md). Приложение читает настройки из файла `.env` в каталоге приложения (или из окружения процесса).

Основные переменные (см. также `backend/.env.example`):

| Переменная       | Описание |
|------------------|----------|
| `PG_HOST`, `PG_PORT`, `PG_DATABASE`, `PG_USER`, `PG_PASSWORD` | Подключение к PostgreSQL |
| `JWT_SECRET`     | Секрет для подписи JWT (задайте надёжную случайную строку в проде) |
| `JWT_EXPIRES_IN` | Срок жизни токена (например `7d`) |
| `PORT`           | Порт HTTP-сервера (например 3000) |
| `PIXEL_OFFICE_WEBHOOK_URL` | Опционально: URL вебхука при создании задачи |

## Учётные записи и роли

- Учётные записи создаются скриптом начального заполнения БД (например `backend/scripts/seed.js`) или через API регистрации (если доступен).
- Роли: **admin**, **manager**, **user**. Разграничение прав:
  - **admin** — полный доступ ко всем задачам и действиям.
  - **manager** — полный доступ ко всем задачам.
  - **user** — доступ только к задачам, где пользователь является автором (creator) или исполнителем (assignee): просмотр, изменение, удаление, запуск/остановка таймера только по ним.

Подробнее об учётках и формате логинов/паролей: [ACCOUNTS.md](../ACCOUNTS.md).

## Схема БД и миграции

- Актуальная схема: `backend/schema.sql`. При первом развёртывании применяется целиком:  
  `sudo cat /home/tasktime/app/backend/schema.sql | sudo -u postgres psql -d tasktime`
- При обновлении с добавлением новых таблиц (например `audit_log`) повторное применение того же файла безопасно: используются `CREATE TABLE IF NOT EXISTS` и `CREATE INDEX IF NOT EXISTS`. Для существующих инстансов достаточно выполнить тот же `schema.sql` (новые объекты создадутся, существующие не изменятся).
- Отдельные миграции (ALTER и т.п.) при необходимости выполняйте вручную и документируйте.

## Резервное копирование

- Рекомендуется регулярно создавать резервные копии базы PostgreSQL, например:
  ```bash
  sudo -u postgres pg_dump -Fc tasktime > /backup/tasktime_$(date +%Y%m%d).dump
  ```
- Восстановление из дампа:
  ```bash
  sudo -u postgres pg_restore -d tasktime --clean --if-exists /backup/tasktime_YYYYMMDD.dump
  ```
- Храните копии в безопасном месте. Рекомендуется автоматизировать резервное копирование (cron) и периодически проверять восстановление.

Подробнее о деплое и командах: [DEPLOY.md](../DEPLOY.md).

## Журнал аудита и логи

- Приложение пишет события в таблицу **audit_log** (логин, регистрация, создание/изменение/удаление задач, старт/стоп таймера). Поля: дата/время, пользователь, действие, сущность, уровень, детали, IP, User-Agent.
- Логи приложения (stdout/stderr) при запуске через systemd смотреть так:  
  `sudo journalctl -u tasktime -f`
- Для передачи событий в SIEM можно экспортировать данные из `audit_log` (например выгрузкой или стримингом); формат и интеграция описываются отдельно при необходимости.

## HTTPS и безопасность на деплое

- Доступ к приложению из интернета должен осуществляться **только по HTTPS** (ТЗ: защита при передаче по открытым каналам, TLS).
- Разверните перед приложением обратный прокси (nginx, Caddy и т.п.) с TLS 1.2 и выше. Настройте сертификат (например Let's Encrypt) и перенаправление HTTP → HTTPS.
- Секреты (JWT_SECRET, пароли БД) не передавайте по незашифрованным каналам и не храните в открытом виде в репозитории. Используйте `.env` с ограниченными правами доступа.

## Обновления

- **Автодеплой (основной способ):** любой `git push origin main` автоматически запускает деплой через GitHub Actions. Статус — в репозитории во вкладке Actions. Лог деплоя на сервере: `/var/log/tasktime-deploy.log`.
- **Ручной деплой (при необходимости):** `sudo -u tasktime bash /home/tasktime/deploy.sh` на сервере.
- После обновления при необходимости примените изменения схемы БД (запуск `schema.sql` или отдельные миграции — вручную, т.к. миграции не автоматизированы).
- Рекомендуется применять обновления безопасности зависимостей (`npm audit`, обновление пакетов) в рамках регламента обслуживания.
- Подробности настройки автодеплоя (SSH-ключ, GitHub Secrets, sudoers): [DEPLOY.md](../DEPLOY.md) раздел «Деплой обновлений».
