# Подключение GitLab webhook к TaskTime — пошагово

Пошаговая инструкция: зачем каждый шаг и что именно делать.

---

## Что вы в итоге получите

GitLab будет сам сообщать TaskTime о событиях в репозитории. TaskTime по ключам задач (например `DEMO-42`) будет менять статусы:

- **Push** в ветку с таким ключом → задача «В работе» (IN_PROGRESS)
- **Merge request открыт** → задача «На ревью» (REVIEW)
- **Merge request влит** → задача «Готово» (DONE)

Чтобы это работало, нужно один раз настроить webhook в GitLab и секрет в TaskTime.

---

## Шаг 0. Убедиться, что backend доступен из интернета

**Зачем:** GitLab слает запросы на ваш сервер. Если backend только на `localhost`, GitLab до него не достучится.

**Что делать:**

- Если TaskTime уже на сервере (staging/production) — запомните его URL, например `https://api.tasktime.example.com`. Это и есть «ваш backend».
- Если пока работаете локально — поднимите туннель, чтобы извне был доступ на ваш компьютер:
  - [ngrok](https://ngrok.com): `ngrok http 3000` → скопируйте выданный URL (типа `https://abc123.ngrok.io`).
  - Или другой туннель (cloudflared, localtunnel и т.п.).

**Итог:** у вас есть URL, по которому из интернета доступен ваш backend. Например: `https://abc123.ngrok.io` или `https://api.tasktime.example.com`.

---

## Шаг 1. Открыть настройки Webhooks в GitLab

**Зачем:** Webhook — это «адрес, куда GitLab будет слать уведомления». Его задают в настройках проекта.

**Что делать:**

1. Зайдите в **ваш проект** в GitLab (тот репозиторий, с которым хотите связать задачи TaskTime).
2. В левом меню: **Settings** (или **Configure**) → **Webhooks**.  
   Либо: **Settings** → **Integrations** → вкладка **Webhooks**.
3. У вас откроется форма «Add new webhook» / «New webhook».

**Итог:** вы видите форму с полями URL, Secret token и списком событий.

---

## Шаг 2. Указать URL webhook

**Зачем:** GitLab должен знать, куда отправлять POST-запросы при push и merge request.

**Что делать:**

В поле **URL** введите (подставьте свой хост вместо `<ваш-backend>`):

```
https://<ваш-backend>/api/integrations/gitlab/webhook
```

Примеры:

- Локально через ngrok: `https://abc123.ngrok.io/api/integrations/gitlab/webhook`
- Ваш сервер: `https://api.tasktime.example.com/api/integrations/gitlab/webhook`

**Итог:** в поле URL — полный адрес до `/api/integrations/gitlab/webhook`. Пока кнопку «Add webhook» не нажимайте.

---

## Шаг 3. Задать Secret token в GitLab

**Зачем:** Секрет нужен, чтобы только ваш GitLab мог вызывать ваш backend. Backend будет сверять присланный токен с тем, что записан у вас в `.env`.

**Что делать:**

1. Сгенерируйте случайную строку. В терминале:
   ```bash
   openssl rand -hex 24
   ```
   Скопируйте вывод (длинная строка без пробелов).
2. В форме webhook в GitLab найдите поле **Secret token** (или **Secret**).
3. Вставьте туда эту строку. **Сохраните её ещё раз в блокнот** — она понадобится в следующем шаге для backend.

**Итог:** в поле Secret token в GitLab — ваша сгенерированная строка, и вы её где-то сохранили.

---

## Шаг 4. Выбрать события (Merge request и Push)

**Зачем:** Мы обновляем статусы только при этих двух типах событий. Остальные можно не включать.

**Что делать:**

В форме webhook отметьте галочки:

- **Merge request events** — когда MR создают, вливают, закрывают.
- **Push events** — когда пушат коммиты в ветку.

Остальные (Tag push, Issue и т.д.) для этой интеграции не обязательны.

**Итог:** включены только Merge request events и Push events.

---

## Шаг 5. Сохранить webhook в GitLab

**Зачем:** До сохранения GitLab не будет ничего отправлять.

**Что делать:**

Нажмите **Add webhook** / **Save** (внизу формы).

После сохранения GitLab покажет webhook в списке. Может сразу отправить тестовый запрос — если backend ещё не настроен или недоступен, в «Recent deliveries» будет ошибка. Это нормально, исправим на следующих шагах.

**Итог:** webhook в GitLab создан и сохранён.

---

## Шаг 6. Добавить тот же секрет в backend (TaskTime)

**Зачем:** Backend проверяет заголовок `X-Gitlab-Token`: если в запросе пришёл тот же токен, что и в переменной окружения, запрос считается подлинным.

**Что делать:**

1. Откройте файл `.env` в папке **backend** вашего проекта TaskTime (рядом с `package.json`).
2. Добавьте строку (подставьте тот же Secret token, что в шаге 3):

   ```bash
   GITLAB_WEBHOOK_SECRET=сюда_вставьте_скопированный_секрет
   ```

   Без кавычек, если в секрете нет пробелов. Если используете кавычки: `GITLAB_WEBHOOK_SECRET="ваш секрет"`.
3. Сохраните файл.
4. Перезапустите backend (остановите и снова запустите `npm run dev` или как вы его обычно запускаете).

**Итог:** в backend в `.env` есть `GITLAB_WEBHOOK_SECRET` с тем же значением, что и в GitLab; backend перезапущен.

---

## Шаг 7. Проверить, что запросы доходят

**Зачем:** Убедиться, что GitLab успешно стучится в TaskTime и что секрет совпадает.

**Что делать:**

1. В GitLab снова откройте **Settings** → **Webhooks** и найдите ваш только что созданный webhook.
2. Внизу страницы есть блок **Recent deliveries** (последние отправки).
3. Нажмите **Test** → выберите, например, **Merge requests events**. GitLab отправит тестовый payload на ваш URL.
4. Посмотрите результат:
   - **HTTP 200** и в ответе что-то вроде `{"ok":true}` — всё ок, webhook работает.
   - **HTTP 401** — неверный или отсутствующий Secret token (проверьте, что в `.env` точно такой же, как в GitLab, и что backend перезапущен).
   - **Failed to connect** / таймаут — до backend из интернета не достучаться (проверьте туннель или хост и порт).

**Итог:** при тесте вы получаете 200 и `ok: true`.

---

## Шаг 8. Как TaskTime понимает, какую задачу обновлять (ключи задач)

**Зачем:** Чтобы не путаться, когда вы пушите код или открываете MR.

**Как это устроено:**

У каждой задачи в TaskTime есть ключ: **код проекта + номер**, например `DEMO-42`, `BACK-1`. Этот ключ TaskTime ищет в:

- имени ветки (например `feature/DEMO-42-fix-bug`);
- названии merge request (например «DEMO-42: починить баг»);
- описании merge request;
- тексте коммита (например «Fix DEMO-42: описание»).

Если ключ найден и такая задача есть в TaskTime, её статус обновится по правилам из таблицы ниже.

**Что делать с вашей стороны:** называйте ветки и MR так, чтобы в них фигурировал ключ задачи (например `DEMO-42`), тогда статусы будут подтягиваться автоматически.

---

## Краткая сводка: что на каком шаге делаете

| Шаг | Где | Что делаете |
|-----|-----|-------------|
| 0 | Компьютер/сервер | Убеждаетесь, что backend доступен из интернета (URL или туннель). |
| 1 | GitLab | Открываете проект → Settings → Webhooks. |
| 2 | GitLab | Вводите URL: `https://<ваш-backend>/api/integrations/gitlab/webhook`. |
| 3 | GitLab + терминал | Генерируете секрет (`openssl rand -hex 24`), вставляете в поле Secret token в GitLab, копируете для шага 6. |
| 4 | GitLab | Включаете события Merge request events и Push events. |
| 5 | GitLab | Нажимаете Add webhook / Save. |
| 6 | TaskTime backend | В `.env` добавляете `GITLAB_WEBHOOK_SECRET=<тот же секрет>`, перезапускаете backend. |
| 7 | GitLab | Test → Merge requests events → смотрите 200 и `ok: true` в Recent deliveries. |

После этого при push и merge request с ключом задачи (например `DEMO-42`) статусы в TaskTime будут обновляться автоматически, а в аудите будет указан источник `GITLAB`.
