# Границы MVP TaskTime

**Для кого:** ИТ-менеджер, продукт, заказчик.  
**Цель:** Понятно зафиксировать, что входит в первую поставку, что отложено и почему упрощение модели ускоряет выход продукта.

---

## Что входит в MVP

В первой версии TaskTime реализованы **пять основных сущностей**:

1. **Пользователи** — учётные записи, вход в систему, роли (например администратор, менеджер, исполнитель).
2. **Проекты** — контейнеры работ: у каждого проекта есть название, описание и список задач.
3. **Задачи (issues)** — единицы работы внутри проекта: заголовок, описание, статус, кто создал, кто исполнитель, приоритет и порядок. Статус задачи может быть только один из четырёх: «открыта», «в работе», «на проверке», «выполнена».
4. **Комментарии** — обсуждения по задаче: кто написал, текст, дата. Все комментарии привязаны к задаче.
5. **Учёт времени (time_logs)** — списание времени по задаче: кто списал, сколько часов, при необходимости краткая заметка.

Логика продукта простая: пользователи работают в проектах с задачами, обсуждают задачи в комментариях и фиксируют затраченное время. Этого достаточно, чтобы вести задачи и контролировать трудозатраты без лишней сложности в первой версии.

---

## Что намеренно отложено

Следующие возможности **не входят в MVP** и запланированы на следующие этапы:

- **Организации** — разведение данных по компаниям/тенантам (мультитенантность). В MVP все пользователи и проекты в одном общем контуре.
- **Метки (labels)** — теги для задач (например по типу работы или срочности). В MVP для классификации достаточно приоритета и статуса.
- **Расширенные доски** — несколько досок на проект, настраиваемые колонки, лимиты «работы в процессе» (WIP). В MVP достаточно одной простой доски/списка с четырьмя фиксированными статусами.
- **Иерархия задач** — связи «родитель — подзадача» (эпик → история → подзадача). В MVP все задачи одного уровня внутри проекта.
- **Владение командами** — команды как отдельные сущности, назначение ответственной команды на проект или задачу. В MVP достаточно назначения задачи на конкретного пользователя.

Эти вещи не забыты: они описаны в архитектурной документации как будущий контур. Решение отложить их принято сознательно, чтобы быстрее получить рабочую систему и не раздувать первый релиз.

## Почему упрощение доменной модели ускоряет поставку

**Меньше сущностей — меньше кода и тестов.**  
Вместо организаций, команд, досок, колонок, меток и иерархии задач в MVP остаются только пользователи, проекты, задачи, комментарии и учёт времени. Меньше таблиц, связей и правил — меньше разработки, меньше сценариев тестирования и меньше мест, где может возникнуть ошибка.

**Фиксированные статусы — проще логика и интерфейс.**  
Четыре статуса (открыта, в работе, на проверке, выполнена) покрывают типичный цикл задачи. Не нужно настраивать статусы по проектам, хранить их в отдельной сущности и согласовывать с колонками досок. Интерфейс и API предсказуемы, доработки быстрее.

**Один уровень задач — проще права и отчёты.**  
Без иерархии не нужно решать, как считать время по родительской задаче, как показывать прогресс по эпику и как разграничивать доступ по уровням. Всё считается и отображается по одной задаче.

**Один контур без организаций — быстрее первый запуск.**  
Мультитенантность добавляет вопросы безопасности, изоляции данных и миграций. В MVP один «тенант» — команда может быстрее выкатить систему, проверить сценарии использования и затем спланировать введение организаций по результатам.

**Итог для ИТ-менеджера:**  
Упрощение модели — это не отказ от «нормальной» системы, а осознанный выбор: сначала закрыть базовый сценарий (задачи, обсуждения, учёт времени) с минимальным риском и сроком, а расширенные возможности добавлять следующими итерациями, когда первая версия уже используется и даёт обратную связь.
