# TaskTime — Предложение по архитектуре (модульный монолит)

**Назначение:** Предложить понятную, удобную для AI структуру бэкенда и приложения, ориентированную на быструю разработку в стиле «вайб-код».  
**Дата:** март 2025

---

## 1. Принципы

- **Модули по фичам** — Код группируется по домену (auth, users, projects, issues и т.д.), а не по слоям (все роуты, все сервисы).
- **Жёсткие границы модулей** — Модули экспортируют небольшой публичный API (сервисы); запрещён кросс-импорт репозиториев и внутренностей.
- **Слои внутри каждого модуля** — API (HTTP) → Service → Repository → БД. DTO на границах.
- **Простота важнее гибкости** — Один стек, один артефакт развёртывания; без микросервисов в MVP.
- **Удобство для AI** — Предсказуемые пути, именование и паттерны, чтобы агенты могли безопасно ориентироваться и менять код.

---

## 2. Список модулей и зоны ответственности

| Модуль | Ответственность | Ключевые сущности |
|--------|-----------------|-------------------|
| **auth** | Вход, регистрация, выход, выдача/проверка JWT, имперсонация, хэш пароля | — |
| **users** | CRUD пользователей, список, роли, блок/разблок; используется auth и admin | User |
| **organizations** | CRUD организаций (опционально в MVP); контур тенанта | Organization |
| **teams** | CRUD команд, добавление/удаление участников | Team, TeamMember |
| **projects** | CRUD проектов, настройки; точка входа для досок/бэклога | Project |
| **issues** | CRUD задач, смена статуса, исполнитель, иерархия (родитель/потомок), список/фильтр | Issue |
| **comments** | CRUD комментариев по задачам | Comment |
| **boards** | Определение доски и колонок; соответствие статусов колонкам | Board, BoardColumn |
| **sprints** | CRUD спринтов, старт/закрытие; назначение задач на спринт | Sprint |
| **time** | Старт/стоп учёта времени, список по задаче/пользователю | TimeLog |
| **notifications** | Заглушка на будущее (in-app, email); no-op или stub | — |
| **audit** | Только добавление в журнал аудита (используется другими модулями) | AuditLog |
| **admin** | Статистика, список пользователей, журнал активности, лог деплоев (чтение audit, users) | — |

Общая инфраструктура (пул БД, env, логгер) выносится в **shared** или **core**, вне модулей.

## 3. Слои (в каждом модуле)

Структура модуля:

```
module/
  api.js          # Express-маршруты: разбор запроса, вызов сервиса, ответ
  service.js      # Бизнес-логика: оркестрация, валидация, права
  repository.js   # Доступ к БД: только запросы, без бизнес-правил
  dto.js          # Формы запроса/ответа (опционально; JSDoc + объекты)
```

- **API** — Только HTTP. Валидация ввода (body/query/params), вызов сервиса, маппинг в JSON. Без SQL.
- **Service** — Оркестрирует репозитории и другие сервисы; проверяет права (или делегирует общему auth-хелперу); возвращает объекты домена или DTO.
- **Repository** — Функции с SQL, возвращают строки/id; без знания HTTP и ролей.
- **DTO** — Опциональный файл с JSDoc-типами или фабриками запроса/ответа; контракты явные для AI и документации.

**Правила зависимостей:**

- API → только Service (внутри модуля).
- Service → Repository (внутри модуля); Service → другие Services (допускается, без глубоких цепочек).
- Repository → только пул БД (общий); Repository → Service нет.
- Кросс-модульно только через Service (например issues.service может вызывать comments.service.listByIssue(issueId)); импорт репозиториев и внутренностей других модулей запрещён.

## 5. Правила зависимостей (сводка)

1. Модули **не импортируют** репозитории или api других модулей. При необходимости вызывают **сервисы** других модулей.
2. **Shared** — единственное место для сквозного кода: auth middleware, audit, errors, config, db.
3. **server.js** подключает маршруты: `app.use('/api/auth', authApi)` и т.д. Каждый модуль экспортирует Router из api.js.
4. **Циклических зависимостей сервисов** нет. Если A и B нужны друг другу — вынести общую логику в третий модуль или shared.
5. Репозитории получают `db` (или `query`/`getClient`) аргументом или общим импортом; без прямого require другого модуля.

---

## 6. Именование

- **Файлы:** api.js, service.js, repository.js, dto.js в модуле; kebab-case для новых многословных файлов.
- **Функции:** getById, list, create, update, remove в репозитории; в сервисе — по сценарию: createIssue, moveIssueToSprint.
- **Маршруты:** REST, ресурс во множественном числе, id в пути.
- **Ошибки:** Общие errors.NotFound(), errors.Forbidden() и т.д. и глобальный обработчик → 404/403/400/500.

---

## 7. Польза для AI-разработки

- **Локальность:** «Добавить поле в Issue» → один модуль (issues), три файла (api, service, repository); изменение схемы в одном месте.
- **Предсказуемость:** Одинаковые слои везде; агенты понимают, куда добавлять код.
- **Контракты:** DTO или JSDoc на границе API/сервиса делают форму запроса/ответа явной для промптов и тестов.
- **Безопасный рефакторинг:** Перенос маршрута из монолита в модуль — вырезать/вставить и поправить импорты; при соблюдении правил скрытых зависимостей нет.

Эта структура — основа для плана реструктуризации репозитория и плана миграции.
