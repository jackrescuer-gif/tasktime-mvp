## 2. Сводка по сущностям

| Сущность | Назначение | Основные связи |
|----------|------------|----------------|
| **users** | Учётные записи, роли, блокировка | Ссылки из tasks, task_items, projects, teams, time_logs, audit_log |
| **tasks** | Плоские «наследие»-задачи | project_id, assignee_id, creator_id; связь с task_items через task_item_links |
| **time_logs** | Учёт времени | task_id (обязательно), task_item_id (опционально), user_id |
| **audit_log** | Аудит / SIEM | user_id, entity_type, entity_id, details (JSONB) |
| **projects** | Контейнер проектов | owner_id; task_items с context_type='project', context_id=projects.id |
| **product_teams** | Контейнер команд | lead_id; участники в product_team_members; task_items context_type='product_team' |
| **product_team_members** | Участники команды | team_id, user_id, role |
| **business_functions** | Метка контекста «быстрых» задач | task_items context_type='quick', context_id=business_functions.id (полиморфно) |
| **task_items** | Иерархическая работа (epic/story/subtask) | parent_id (самоссылка), context_type/context_id (project/quick/product_team) |
| **task_item_links** | Связь плоской задачи ↔ иерархического элемента | task_id, task_item_id, link_type |

## 3. Отсутствующие сущности (для Jira-подобного MVP)

| Отсутствует | Влияние |
|-------------|---------|
| **organizations** | Нет мультитенантности; все данные глобальные. Позже сложно добавить настройки/биллинг/изоляцию на уровне организации. |
| **comments** | Нет обсуждения по задачам/тикетам. Нужно для совместной работы. |
| **sprints** | Нет сущности спринта; нельзя назначить задачи на временной бокс или показать бэклог спринта. |
| **boards** | Нет сущности доски; «доска» неявная (например вид проекта по статусу). Нельзя иметь несколько досок на проект. |
| **board_columns** | Статус — свободная строка/enum в коде; нет колонок с порядком и WIP-лимитами. |
| **labels** | Нет таблицы меток; нельзя тегировать задачи несколькими метками. |
| **statuses** | Статус — строка/CHECK в колонках; не отдельная сущность. Разные workflow по проектам/доскам потребуют новой схемы. |
| **issue_history** | Нет истории изменений полей (кто когда сменил статус и т.д.); только audit_log для высокоуровневых действий. |

## 4. Рисковые решения в схеме

- **Двойная модель задач (tasks + task_items):** Два способа представления работы с опциональной связью. Логика размазана (например «создать стори из задачи»), RBAC и API должны учитывать оба. Объединение в одну модель «Issue» упростит домен и работу с AI.
- **Полиморфный контекст (task_items):** `context_type` + `context_id` указывают на project, product_team или business_functions. Нет FK на одну таблицу; сложнее обеспечивать целостность и делать запросы «все элементы проекта» одним джойном. Для MVP яснее один `project_id` (nullable) и опционально `team_id`.
- **tasks.project_id добавлен после projects:** Порядок в схеме: сначала tasks, потом projects, затем ALTER tasks ADD project_id. Риск циклических ссылок снят порядком создания, но порядок миграций и инициализации хрупкий.
- **Двойная ссылка в time_logs:** Может быть привязка к `task_id` и/или `task_item_id`. Неоднозначность: что считать каноническим для записи? Одна сущность Issue уберёт это.
- **Роль в users:** Одна строка (admin, manager, user, cio, viewer, super-admin). Нет ролей по проекту/команде; масштабирование до прав на уровне проекта потребует новой таблицы (например project_members с role).
- **Нет мягкого удаления:** DELETE — жёсткое. Восстановление или состояние «архив» потребуют новых колонок или таблиц.

---

## 5. Нормализация

- **Денормализованные имена в ответах:** API часто возвращает assignee_name, creator_name, project_name, lead_name через JOIN. В БД не дублируются; для чтения приемлемо. Нет материализованных агрегатов (например «стори по проекту» считаются на лету).
- **task_items:** Нормализованы хорошо; context_type/context_id полиморфны, но не избыточны. Порядок в контексте — order_index в той же таблице.
- **audit_log.details JSONB:** Гибко, но без типов и схемы. Удобно для расширяемости; неудобно для строгих запросов/аналитики без соглашений.
- **product_team_members.role:** Свободная строка (lead, member). Не нормализовано в таблицу ролей; для MVP допустимо.

---

## 6. Использование индексов (текущее)

- **tasks:** assignee_id, creator_id, status, project_id — для списков и фильтров.
- **time_logs:** task_id, user_id — для «логи по задаче» и «мои логи».
- **audit_log:** created_at, user_id, (entity_type, entity_id), level — для активности и админ-представлений.
- **task_items:** parent_id, (context_type, context_id), assignee_id, status — для дерева и списков по контексту.
- **task_item_links:** task_id, task_item_id — для разрешения связей.

Для масштабирования позже не хватает составных индексов под «открытые задачи в проекте по order_index», «задачи в спринте по колонке» и т.п., когда появятся спринты и доски.

---

## 7. Вывод

Текущая схема поддерживает проекты, продуктовые команды, плоские задачи, иерархические task_items, учёт времени и аудит. Для Jira-подобного MVP основные пробелы: **organizations**, **comments**, **sprints**, **boards/columns**, **labels**, **issue history**. **Двойная модель задач** и **полиморфный контекст** — главные архитектурные риски для учёта в доменной модели MVP и миграции (см. MVP_DOMAIN_MODEL.md и MIGRATION_PLAN.md).
