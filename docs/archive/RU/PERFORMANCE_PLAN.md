# Flow Universe — Стратегия производительности

**Назначение:** Стратегии для больших досок, большого числа задач, кэширования и пагинации. Фокус на MVP; без переусложнения.  
**Дата:** март 2025

---

## 1. Большие доски и много задач

**Проблема:** Доска с множеством колонок и сотнями задач — один ответ «все задачи проекта» может быть тяжёлым и медленным. DOM с сотнями карточек — тормоза при раскладке и drag-and-drop. Списки (бэклог, список задач) — медленные при неограниченной выборке.

**Бэкенд:** Индексы по (project_id, status), (project_id, sprint_id), (assignee_id, status). Все списковые эндпоинты с limit и offset (или page и limit). Лимит по умолчанию (например 50), максимум (например 200). Загрузка доски: один GET с фильтрами с limit (например 500) или по одному запросу на колонку при ленивой подгрузке тяжёлых колонок. В списках возвращать только поля, нужные для карточек (id, title, status, assignee_id, order_index, priority, type); полное тело — только для GET /api/issues/:id.

**Фронтенд:** Для длинных списков (бэклог, колонка со 100+ карточками) — виртуальный список (рендер только видимых элементов). В MVP можно начать с пагинации. По доске: в колонке с большим числом задач показывать первые N (например 20) и «Показать ещё» или подгрузку при раскрытии. Дебаунс поиска/фильтра (например 300 мс) перед запросом к API.

---

## 2. Кэширование

**HTTP (API):** Для GET списков/доски — Cache-Control: private, max-age=0 или no-store, чтобы клиент не кэшировал устаревшее. Опционально короткий max-age для GET списка задач, если нужна разгрузка; тогда инвалидация при мутациях сложнее. Для MVP — без кэша или короткий приватный кэш.

**Клиент:** В памяти хранить последнюю загрузку доски/списка; использовать до смены раздела или явного «Обновить». При любой мутации (создание, обновление, удаление, перемещение) — инвалидировать или точечно обновить кэш. Крупные списки в localStorage не хранить (лимиты и устаревание). Оптимистичное обновление уже уменьшает ощущаемую задержку без серверного кэша.

**Сервер (опционально позже):** Redis/память для «состояния доски» (список id по колонкам) с коротким TTL (30 с) и инвалидация при обновлении любой задачи проекта. Для MVP не обязательно при наличии индексов.

---

## 3. Пагинация

**Контракт:** Параметры limit (по умолчанию 20, макс 100), offset (по умолчанию 0) или page и limit. Ответ: массив элементов; опционально общее количество в заголовке (X-Total-Count) или в теле ({ items: [], total: N }). Если total дорогой — можно не возвращать или вынести в отдельный эндпоинт (GET /api/issues/count?project_id=…).

**Где применять:** GET /api/issues (или /api/projects/:id/issues) — с пагинацией. GET /api/projects — при большом числе проектов. GET /api/admin/activity — уже limit/offset. Бэклог — тот же список с sprint_id=null, с пагинацией. Доска: либо одна страница «всех задач доски» (и принять частичные колонки), либо подгрузка по колонкам с limit. Для MVP предпочтительно один запрос с limit (например до 200 задач) и «Загрузить ещё» в UI при необходимости.

**Сортировка:** Стабильная: order_index, затем id (или created_at), чтобы пагинация не «прыгала» при новых вставках. Один и тот же ORDER BY в списках и в логике reorder.

---

## 4. База данных

Индексы — см. DOMAIN_MODEL.md и схему. Минимум: issues(project_id), issues(project_id, status), issues(sprint_id), issues(assignee_id). Избегать N+1: не грузить задачи, а затем по одной — исполнителя; один JOIN или пакетная подгрузка имён в список. Подсчёт total: отдельный лёгкий COUNT(*) с теми же фильтрами или материализованный счётчик при записи; для MVP одного запроса count на список достаточно.

---

## 5. Итог

| Область | Стратегия |
|---------|-----------|
| Большие доски | Лимит задач на запрос; одна выборка по области или по колонкам; виртуализация или «загрузить ещё» в тяжёлых колонках. |
| Много задач | Пагинация (limit/offset или page/limit); стабильная сортировка; индексы по project_id, status, sprint_id. |
| Кэширование | Клиент: последний результат в памяти; инвалидация при мутации. Сервер: опционально короткий TTL для списков; для MVP можно без кэша. |
| Пагинация | Во всех списковых эндпоинтах; опционально total; зафиксировать в API. |

Для MVP достаточно индексов + limit/offset + компактных ответов + опционального «загрузить ещё» в UI. Виртуализацию и серверный кэш добавлять по метрикам.
