# TaskTime — План безопасной поэтапной миграции

**Назначение:** Пофазный переход от текущего монолита к модульной архитектуре и (опционально) к новой доменной модели без поломки работающего MVP.  
**Дата:** март 2025

---

## 1. Принципы

- **Без «большого взрыва»:** По одному срезу; каждая фаза развёртываема и тестируема.
- **Обратная совместимость:** Контракты API (пути, форма запроса/ответа) не меняются до явного версионирования или изменения.
- **Рискованные изменения изолированы:** Изменения схемы (например tasks → issues) в отдельных фазах со скриптами миграции и планом отката.
- **Сначала функциональная эквивалентность:** Сначала реструктуризация (выделение модулей), затем новый домен (спринты, доски, комментарии); новые фичи — уже на новой структуре.

---

## 2. Обзор фаз

| Фаза | Цель | Риск | Срок (оценка) |
|------|------|------|----------------|
| **0** | Подготовка: shared auth, errors, audit | Низкий | 1–2 дня |
| **1** | Выделение модулей (auth, users, projects, tasks, time, teams, admin) | Низкий | 1–2 недели |
| **2** | Новый домен: комментарии, спринты, доски/колонки, метки | Средний | 2–3 недели |
| **3** | Объединение tasks + task_items → issues (схема + API) | Высокий | 2–3 недели |
| **4** | Структура фронтенда + опционально организации | Низкий–средний | 1–2 недели |

---

## 3. Фаза 0 — Подготовка

**Цель:** Общая инфраструктура без изменения поведения маршрутов.

**Шаги:** Добавить shared/errors.js (NotFound, Forbidden, BadRequest, Unauthorized); перенести auth middleware и хелперы ролей в shared/auth.js; перенести audit в shared/audit.js; добавить config.js из process.env. Проверка: все текущие тесты и ручной сценарий (вход, список задач, создание, таймер). Откат: отмена коммитов; БД и контракт API не менялись.

---

## 4. Фаза 1 — Выделение модулей (контракт не меняется)

**Цель:** Обработчики и логика в модулях по фичам; server.js только подключает роутеры. Те же URL и JSON.

**Порядок выделения:** auth → users → projects → tasks (все /api/tasks, task-items, task-links) → time → teams → admin (+ audit для чтения). По модулю: создать api, service, repository; скопировать обработчики; SQL в репозиторий, права в сервис; убрать дубликаты из server.js, подключить роутер. Проверка: полный регресс. Откат: отмена коммита модуля, восстановление обработчиков в server.js.

---

## 5. Фаза 2 — Новый домен (Comments, Sprints, Boards, Labels)

**Цель:** Новые сущности и API без удаления существующих. Схема БД только добавления.

**Шаги:** Добавить таблицы comments, sprints, boards, board_columns, labels, issue_labels (см. MVP_DOMAIN_MODEL.md). Пока оставить tasks и task_items; при необходимости связать новые таблицы с ними (или с issue_id после фазы 3). Новые модули: comments, sprints, boards, метки; соответствующие эндпоинты. Проверка: новые эндпоинты работают, старые без изменений. Откат: удаление новых таблиц и маршрутов; потери данных в существующих таблицах нет.

---

## 6. Фаза 3 — Объединение tasks + task_items → issues (высокий риск)

**Цель:** Одна модель и таблица Issue; миграция данных из tasks и task_items; упразднение двойного API или маппинг на issue API.

**Риски:** Ошибки миграции данных, простой, поломка фронта. Изолировать флагом фичи или параллельным API.

**Шаги:** Спроектировать таблицу issues; написать идемпотентный скрипт миграции (копирование task_items и tasks в issues, перенос ссылок в time_logs, comments и т.д.); опционально период двойной записи; новые маршруты /api/issues, старые делегировать или помечать устаревшими; фронт перевести на новый API; удалить старые таблицы/связи. Проверка: полный E2E; скрипт отката. Откат: восстановление предыдущей версии приложения и откат миграции при необходимости.

---

## 7. Фаза 4 — Фронтенд и опционально организации

**Цель:** Упорядочить фронтенд; при необходимости добавить организации (мультитенантность). Разбить app.html на логические файлы или ввести минимальную сборку; опционально org_id в users, projects, teams. Проверка: все сценарии работают.

---
