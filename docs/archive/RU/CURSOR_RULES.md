# Flow Universe — Правила разработки для Cursor AI

**Назначение:** Правила для повышения продуктивности AI (Cursor): именование, структура, типизация, API, компоненты, доступ к БД. Использовать как справочник и при необходимости объединять с `.cursor/rules/` или cursorrules.  
**Дата:** март 2025

---

## 1. Именование

### Бэкенд (Node/Express)

- **Файлы:** api.js, service.js, repository.js в модуле; kebab-case.js для многословных хелперов.
- **Функции (репозиторий):** getById(id), list(filters), create(data), update(id, data), remove(id). Одна сущность на репозиторий.
- **Функции (сервис):** глагол + сущность: createIssue, updateIssueStatus, listIssuesByProject, addCommentToIssue.
- **Маршруты:** REST, ресурс во множественном числе, id в пути.
- **Переменные:** camelCase. Константы (env, config): UPPER_SNAKE или camelCase в объекте config.
- **Запрос/ответ:** Форма как в docs/API.md; в JSON snake_case для согласования с БД (или camelCase для фронта — выбрать один вариант и зафиксировать).

### Фронтенд (Vanilla JS / будущий фреймворк)

- **ID и классы:** kebab-case (#main-dashboard, .issue-card).
- **data-атрибуты:** data-* для состояния: data-issue-id, data-column-id.
- **Функции:** camelCase: loadIssues(), renderBoard(), openIssueModal(id).
- **Клиент API:** Один хелпер apiFetch(path, options); path всегда начинается с /api/.

### База данных

- **Таблицы:** snake_case, множественное число: users, issues, board_columns.
- **Колонки:** snake_case: created_at, assignee_id, project_id.
- **Индексы:** idx_<таблица>_<колонки>: idx_issues_project_id, idx_issues_sprint_id.

---

## 2. Структура каталогов

Бэкенд после реструктуризации: server.js, db.js, config.js, shared/ (auth, audit, errors), modules/<модуль>/ (api.js, service.js, repository.js), scripts, schema.sql. Один модуль = одна фича; не импортировать repository другого модуля (только сервис).

Фронтенд: index.html, app.html, admin.html; опционально js/, js/features/. При добавлении фичи предпочтительно отдельный скрипт в js/features/ или секция в app.html; избегать одного файла на 3000+ строк.

---

## 3. Типизация и контракты

TypeScript для MVP не обязателен. JSDoc для публичного API. В сервисе документировать параметры и возвращаемое значение. В слое API — ожидаемое тело и ответ (комментарий или docs/API.md); валидация запроса в одном месте.

---

## 4. Контракты API

Базовый URL: /api. Версии в пути пока нет; добавить /v1 при ломающих изменениях. Авторизация: все маршруты кроме /api/auth/login, /api/auth/register, /health требуют Authorization: Bearer <JWT> или валидную HttpOnly cookie. Успех: 200/201 с JSON. Ошибка: 4xx/5xx с JSON { error: "сообщение" }. Списки: поддержка ?limit= и ?offset= (или ?page= и ?limit=); зафиксировать в API.md. Роль и чувствительные поля из тела запроса не принимать; только req.user из JWT.

---

## 5. Паттерны компонентов (фронтенд)

Одна фича — одна функция отрисовки (renderBoard, renderIssueList, renderIssueDetail). Модалки: один контейнер, показ/скрытие по классу или атрибуту; контент по текущему выбору. Списки: один контейнер, innerHTML или шаблон; при больших объёмах — пагинация или виртуальный скролл. Состояние минимальное: текущий пользователь, текущий проект/доска, id открытой модалки. После мутации предпочтительно повторный запрос (например loadBoard()) кроме явного оптимистичного обновления.

---

## 6. Доступ к БД

Всегда использовать общий пул (require('../db') или переданный из приложения). Не создавать новый pg.Pool в модуле. SQL только в repository.js; не в api.js и service.js. Сервис вызывает репозиторий и при необходимости другие сервисы. Запросы только параметризованные. Изменения схемы — в schema.sql, идемпотентные. При каждом создании/обновлении/удалении сущности вызывать audit({ userId, action, entityType, entityId, req }).

---

## 7. Безопасность (кратко)

Auth: проверка JWT (или cookie) в middleware, установка req.user. Роль и user_id из тела не доверять. RBAC: проверка в слое сервиса (или общем хелпере). Ввод: проверка обязательных полей и типов в API; при невалидном — 400.

---

## 8. Чего не делать (для AI)

Не вводить новый ORM и не менять драйвер БД без оформленного решения. Не добавлять кросс-модульный импорт репозиториев; только сервис к сервису. Не дублировать логику auth и audit; использовать общий middleware и audit(). Не менять форму ответа API без обновления docs/API.md и фронта. Не деплоить без явного подтверждения пользователя (по правилам проекта).

Полный текст с примерами кода — в docs/ENG/architecture/CURSOR_RULES.md.
