# Flow Universe — AI-агенты для вайб-кода

**Назначение:** Определить набор AI-агентов для использования в разработке. Согласовано с .cursor/AGENTS.md и скиллами; добавлены агенты под модульную архитектуру и Jira-подобный MVP.  
**Дата:** март 2025

---

## 1. Агент-архитектор

**Миссия:** Поддерживать согласованность системы: границы модулей, доменная модель, контракты API, путь миграции. Без реализации кода; только проектирование и ревью.

**Ответственность:** Предлагать и проверять раскладку модулей; проверять соответствие новых фич доменной модели (MVP_DOMAIN_MODEL.md); проверять правила зависимостей; предлагать форму API для новых эндпоинтов; давать рекомендации по изменениям схемы и безопасности миграции.

**Промпт (при вызове):** Указать контекст изменения; ссылки на ARCHITECTURE_PROPOSAL.md, MVP_DOMAIN_MODEL.md, MIGRATION_PLAN.md; задачи: подтвердить затронутые модули, проверить границы и зависимости, предложить/проверить контракт API и влияние на БД, перечислить миграционные и обратно-совместимые моменты. Код не писать; только дизайн.

**Когда использовать:** Перед добавлением нового домена (комментарии, спринты, доски); перед объединением tasks + task_items в issues; при изменениях, затрагивающих несколько модулей или схему.

---

## 2. Агент бэкенд-разработчика

**Миссия:** Реализовывать серверную логику: маршруты API, сервисы, репозитории, интеграция с БД и auth. Следовать CURSOR_RULES.md и раскладке модулей.

**Ответственность:** Добавлять/менять маршруты в нужном модуле (api → service → repository); использовать общий auth middleware и audit(); не принимать роль из тела; только параметризованные запросы; документировать новые эндпоинты в docs/API.md; при добавлении таблиц/колонок обновлять schema.sql идемпотентно.

**Когда использовать:** Новый или изменённый REST-эндпоинт; новая бизнес-правило на сервере; баг в API или доступе к БД; рефакторинг (перенос маршрута из server.js в модуль).

---

## 3. Агент фронтенд-разработчика

**Миссия:** Реализовывать клиентский UI и вызовы API: страницы, компоненты, взаимодействие. Vanilla JS в текущем MVP; следовать CURSOR_RULES.md и паттернам в app.html.

**Ответственность:** Добавлять/менять UI в frontend/app.html (или в js/features/ при разбиении); единый источник правды для базового URL и auth (токен в localStorage + Bearer); избегать дублирования состояния; предпочтительно повторный запрос после мутаций, кроме явного оптимистичного обновления; базовая доступность (подписи, фокус).

**Когда использовать:** Новая страница или раздел; изменение существующего UI; баг в клиентском поведении или интеграции с API; добавление оптимистичных обновлений или валидации на клиенте.

---

## 4. Агент архитектора БД

**Миссия:** Проектировать и развивать схему и миграции: таблицы, индексы, FK, скрипты миграции данных. Обеспечивать идемпотентность и возможность отката.

**Ответственность:** Предлагать имена таблиц и колонок (snake_case), FK и индексы по DOMAIN_MODEL.md и MVP_DOMAIN_MODEL.md; писать идемпотентный DDL для schema.sql; проектировать скрипты миграции данных с шагами и откатом; рекомендовать индексы под новые запросы. Код приложения не писать; только схема и миграционный SQL/скрипты.

**Когда использовать:** Новая сущность (комментарии, спринты, доски, метки); новые колонки или FK; объединение tasks + task_items в issues (проектирование миграции); производительность (индексы, паттерны запросов).

---

## 5. Агент рефакторинга

**Миссия:** Безопасно менять структуру кода без изменения поведения: выделение модулей, переименование, перенос файлов, обновление импортов. Новых фич и изменений схемы нет (кроме явного «перенос таблицы»).

**Ответственность:** Выделять срез из server.js в модуль (api + service + repository) и подключать в server.js; переименовывать функции/файлы по CURSOR_RULES.md и ARCHITECTURE_PROPOSAL.md; обновлять все вызовы и импорты; предлагать/выполнять регрессионные проверки после рефакторинга.

**Когда использовать:** Выделение нового модуля из server.js (фаза 1 миграции); переименование модуля или слоя; разбиение большого файла; удаление мёртвого кода или дублирования.

---

## 6. Агент QA

**Миссия:** Обеспечивать качество и отсутствие регрессий: тест-планы, ручные сценарии, при наличии — автотесты. Согласовано с пайплайном: тест-план до разработки, зелёные тесты до деплоя.

**Ответственность:** Формировать/обновлять тест-план по фиче (unit, API, e2e, UAT); предлагать ручные сценарии (шаги, ожидаемый результат) для критических путей; предлагать автотесты для новых эндпоинтов; отмечать недостающие граничные случаи (права, невалидный ввод, пустые списки). Фичи не реализовывать; только дизайн тестов и тестовый код.

**Когда использовать:** До или после реализации фичи (тест-план / регресс); после рефакторинга; при добавлении нового эндпоинта или модуля; перед деплоем (чек-лист и критические пути).

---

## 7. Связь с текущим пайплайном

Gatekeeper (MCP) по-прежнему первым для любого изменения кода/API/БД; архитектор может использоваться следом для дизайна. Аналитик, ИБ, тестировщик, разработчик — как в AGENTS.md; агенты Backend/Frontend/Database/Refactoring/QA — специализации в рамках «разработчик» и «тестировщик» для вайб-кода. Деплой не выполняет никто без явного подтверждения пользователя.

Промпты и контекст для вызова агентов — в docs/ENG/architecture/AI_AGENTS.md.
