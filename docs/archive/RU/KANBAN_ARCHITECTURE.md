# Flow Universe — Архитектура Kanban-доски

**Назначение:** Проектирование производительной Kanban-доски: перетаскивание, оптимистичные обновления, порядок задач, структура колонок.  
**Дата:** март 2025

---

## 1. Структура колонок

Доска содержит упорядоченные BoardColumns. У каждой колонки: status_key, order_index, опционально wip_limit. Задачи группируются по issue.status === column.status_key (один проект, опционально фильтр по спринту). Порядок внутри колонки — по issue.order_index (или created_at).

**Поток данных:** GET /api/boards/:boardId или /api/projects/:projectId/board → доска + колонки. GET /api/projects/:projectId/issues с фильтром по статусам → задачи. Фронт группирует по статусу и сортирует по order_index; одна колонка на BoardColumn.

---

## 2. Перетаскивание (drag and drop)

**UX:** Пользователь тянет карточку из одной колонки в другую или меняет порядок в той же колонке. Та же колонка — меняется только order_index. Другая колонка — меняются status и order_index в целевой колонке.

**Клиент:** HTML5 Drag and Drop или небольшая библиотека (Sortable.js, dnd-kit). При drop — вызов API обновления статуса и/или переупорядочивания (PATCH /api/issues/:id или POST /api/issues/:id/move; при переупорядочивании в колонке — PATCH /api/issues/reorder).

**Рекомендация:** PATCH /api/issues/:id с { status, order_index } для перемещения; PATCH /api/issues/reorder при массовом изменении порядка в одной колонке.

---

## 3. Оптимистичные обновления

Сразу после drop перемещать карточку в DOM в целевую колонку/позицию. Затем отправлять PATCH (или reorder). При успехе — оставить UI как есть. При ошибке (4xx/5xx) — вернуть карточку в исходную колонку/позицию и показать ошибку. Сохранять копию предыдущего состояния (id → { status, order_index }) для отката. Удаления остаются пессимистичными (подтверждение затем удаление). Конфликты при одновременном редактировании: в MVP last-write-wins; без real-time разрешения конфликтов.

---

## 4. Порядок задач внутри колонки

Поле order_index (целое) у Issue. При перемещении в колонку: установить issue.status и issue.order_index (в конец колонки — max+1 или 0; между карточками — значение между соседями или перенумерация 1,2,3…). Опционально периодическая «уплотнение» индексов (компактная перенумерация). API: одиночный ход — PATCH /api/issues/:id; массовое переупорядочивание — PATCH /api/issues/reorder с { status, issue_ids: [id1, id2, …] }, сервис выставляет order_index по позиции в массиве в одной транзакции.

---

## 5. Производительность (кратко)

Загрузка доски: один запрос доска+колонки и один запрос списка задач в области (проект ± спринт) или по колонкам при ленивой подгрузке. Для MVP предпочтительно один список + группировка по статусу. Пагинация при большом числе задач — см. PERFORMANCE_PLAN.md. Кэш на клиенте для текущей доски/спринта; инвалидация при мутациях. WebSocket в MVP не требуется; обновление по запросу или опросу.

---

## 6. WIP-лимиты (опционально)

BoardColumn.wip_limit — опциональное целое. Фронт может показывать «3 / 5» и предупреждать или блокировать при превышении. В MVP — только отображение; позже в сервисе можно отклонять перемещение в колонку при превышении лимита (400 с сообщением).

---

Итог: колонки из определения доски; drag-and-drop с вызовом PATCH issue или reorder; оптимистичное обновление с откатом при ошибке; порядок через order_index и reorder API; одна загрузка области доски или по колонкам при необходимости.
