## Рабочая область «MVP замены Jira на вайб-коде»

### Что это такое

В TaskTime появилась отдельная рабочая область для реального проекта **«MVP замены Jira на вайб-коде»**.  
Это не демо-сценарий, а живой бэклог задач, связанных с заменой Jira на лёгкий таск‑трекер вокруг TaskTime и ИИ‑агентов.

### Где искать в интерфейсе

- В боковом меню нажмите пункт **«Проекты»**.
- Вверху страницы есть блок **«Живые продуктовые проекты»** (Real project).
- В этом блоке вы увидите карточку проекта **«MVP замены Jira на вайб-коде»** с бейджем **MVP Jira**.
- Ниже расположен отдельный блок **«Демонстрационные проекты»** — это синтетика для примеров, её можно использовать как справочный материал, но не как рабочий бэклог.

Таким образом, реальные задачи всегда визуально отделены от демонстрационных сценариев.

### Как добавлять задачи в проект

Для проекта «MVP замены Jira…» задачи представлены в виде иерархии **эпики → сториз → сабтаски** (как в Jira).

Добавить задачу можно двумя способами:

1. **Через интерфейс эпиков и сториз**
   - Откройте проект «MVP замены Jira…».
   - В нижней части страницы виден «деревянный» список задач (эпики/сториз/сабтаски).
   - Через этот интерфейс (или с помощью ИИ‑агентов) создаются новые элементы и уточняется структура бэклога.

2. **Через ИИ‑агентов**
   - Агенты используют тот же API `task_items` с привязкой к контексту `context_type = 'project'` и `context_id` проекта «MVP замены Jira…».
   - Вся такая активность попадает в тот же Kanban и дерево задач.

### Kanban‑доска проекта

Для «MVP замены Jira…» доступен отдельный Kanban‑борд:

- Откройте карточку проекта **«MVP замены Jira на вайб-коде»**.
- В верхней части страницы появится блок **«Kanban — сториз проекта»** c тремя колонками:
  - **Backlog** — идеи и задачи, которые ещё не в работе.
  - **In progress** — то, что сейчас активно делается.
  - **Done** — завершённые элементы.
- В колонках показываются, в первую очередь, **сториз** проекта (уровень `story`): по ним удобно отслеживать прогресс по крупным шагам.

Перемещение задачи между колонками:

- На карточке сториз в Kanban есть маленькие кнопки:
  - **Backlog** — вернуть задачу в «входящие» (ещё не в работе).
  - **In progress** — перевести в работу.
  - **Done** — отметить как завершённую.
- После клика статус обновляется, и карточка переезжает в нужную колонку.

### Чем реальные и демо‑проекты отличаются визуально

- **Реальный проект (MVP Jira):**
  - На странице «Проекты» расположен в отдельном блоке «Живые продуктовые проекты».
  - Имеет яркий бейдж **MVP Jira** и более акцентную рамку карточки.
  - При открытии карточки появляется пояснение, что это живой проект, и включается Kanban‑доска.

- **Демо‑проекты:**
  - Находятся в блоке «Демонстрационные проекты».
  - Более нейтральное оформление (приглушённый заголовок раздела, бейдж Demo).
  - Используются как демонстрационный материал, а не как основная рабочая область.

### Как это устроено технически (для команды)

- В таблицу `projects` добавлено поле `project_type`:
  - `demo` — синтетические / демонстрационные проекты (значение по умолчанию);
  - `real` — живые продуктовые проекты.
- В скрипте `backend/scripts/seed.js` создан новый проект:
  - `name: 'MVP замены Jira на вайб-коде'`
  - `project_type: 'real'`
  - `business_goal`: формулировка цели MVP (замена Jira на vibe-коде).
- Страница `frontend/app.html`:
  - Делит проекты на две секции по `project_type`.
  - Для `project_type = 'real'` строит Kanban‑доску по сториз (`/api/task-items?context_type=project&context_id=<id>&level=story`) и меняет статус через `PATCH /api/task-items/:id/status`.

### Как использовать в реальной работе

- Все задачи по реальному MVP (архитектура, аналитика, разработка, тесты, деплой, настройки) должны попадать в проект **«MVP замены Jira на вайб-коде»**.
- Демонстрационные проекты **не** используются для реальных задач, чтобы не смешивать учебные примеры и работу.
- Kanban‑доска даёт быстрый обзор:
  - что уже сделано по MVP;
  - какие сториз в работе;
  - что остаётся в бэклоге.

При необходимости область можно расширить дополнительными статусами и swimlane, но для первой итерации достаточно Backlog / In progress / Done.
