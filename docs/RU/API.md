# Интеграционный API Flow Universe

Описание REST API для интеграции с внешними системами (ТЗ ТР.9, ТР.10). API стабилен для использования в интеграциях и может расширяться с версионированием при необходимости.

---

## Базовый URL и аутентификация

- Все запросы к API (кроме входа и регистрации) требуют заголовок:
  ```http
  Authorization: Bearer <JWT>
  ```
- JWT выдаётся при успешном **POST /api/auth/login** (или при регистрации). Срок жизни задаётся на сервере (`JWT_EXPIRES_IN`, по умолчанию 7 дней).
- При отсутствии или невалидном токене сервер возвращает **401 Unauthorized**.

---

## Коды ответов

| Код | Значение |
|-----|----------|
| 200 | OK, тело — данные (объект или массив) |
| 201 | Created, тело — созданный ресурс |
| 204 | No Content (успех без тела, например после DELETE) |
| 400 | Bad Request — неверные параметры или тело запроса |
| 401 | Unauthorized — нет или неверный токен |
| 403 | Forbidden — нет прав на операцию (разграничение по ролям и владельцу задачи) |
| 404 | Not Found — ресурс не найден |
| 409 | Conflict — например, таймер уже запущен по задаче |
| 500 | Internal Server Error — ошибка сервера |

В случае ошибки в теле может быть объект с полем `error` (строка).

## Эндпоинты

### Аутентификация

- **POST /api/auth/register**  
  Тело: `{ "email", "password", "name" [, "role" ] }`.  
  Ответ 201: `{ "user": { id, email, name, role }, "token" }`.  
  Конфликт по email: 409.

- **POST /api/auth/login**  
  Тело: `{ "email", "password" }`.  
  Ответ 200: `{ "user": { id, email, name, role }, "token" }`.  
  Неверные данные: 401.

### Пользователи

- **GET /api/users**  
  Список пользователей (id, email, name, role) для выбора исполнителя и т.п. Требует авторизацию.

### Задачи

- **GET /api/tasks**  
Список задач. Права: для роли `user` возвращаются только задачи, где пользователь — автор или исполнитель; для `admin` и `manager` — все.  
Query: `assignee_id`, `status`, `creator_id`, `project_id` (опционально).  
Ответ: массив объектов с полями `id`, `title`, `description`, `type`, `priority`, `status`, `assignee_id`, `creator_id`, `estimated_hours`, `project_id`, `created_at`, `updated_at`, `assignee_name`, `creator_name`, `project_name`.

- **POST /api/tasks**  
Создание задачи. Тело: `title` (обязательно), `description`, `type`, `priority`, `status`, `assignee_id`, `estimated_hours`, `project_id` (опционально).  
Ответ 201: созданная задача.

- **GET /api/tasks/:id**  
  Одна задача. 403, если у текущего пользователя нет права на просмотр (не автор и не исполнитель и не admin/manager).

- **PUT /api/tasks/:id**  
Обновление задачи. Тело: те же поля, что при создании (частичное обновление, включая `project_id`). 403 при отсутствии права на изменение.

- **DELETE /api/tasks/:id**  
  Удаление задачи. 403 при отсутствии права на удаление.

### Связи задача ↔ стори

- **POST /api/task-links**  
  Создать связь между плоской задачей (`tasks`) и иерархическим элементом (`task_items`, как правило story).  
  Доступно ролям `admin` и `manager`.  
  Тело: `{ "task_id", "task_item_id", "link_type" = "origin" }`.  
  Используется для того, чтобы стори «знала», из какой задачи она выросла, а задача — какие сториз её реализуют.

### Учёт времени

- **POST /api/tasks/:id/time/start**  
  Старт таймера по задаче. Один активный таймер на пользователя. 403, если нет права на просмотр задачи. 409, если таймер по этой задаче уже запущен.

- **POST /api/tasks/:id/time/stop**  
  Остановка таймера по задаче. Ответ: запись time_log (id, task_id, user_id, started_at, ended_at, duration_minutes).

- **GET /api/tasks/:id/time-logs**  
  Список записей времени по задаче. 403, если нет права на просмотр задачи.

- **GET /api/time-logs**  
  Записи времени текущего пользователя. Query: `task_id` (опционально).

## Прочее

- **GET /health**  
  Проверка доступности сервиса. Ответ 200: `{ "ok": true }`. Не требует авторизации.

## Admin API (`/api/admin/*`)

Все эндпоинты требуют авторизации (`authMiddleware`) + роль `admin` или `super-admin` (`adminMiddleware`). При отсутствии роли — 403.

- **GET /api/admin/stats**  
  Показатели доступности системы: аптайм приложения и сервера, Node.js версия, RAM, load avg, статус и задержка БД, число ошибок за 24ч.

- **GET /api/admin/users**  
  Список всех пользователей с полями: `id`, `email`, `name`, `role`, `is_blocked`, `created_at`, `audit_actions`, `last_activity`.

- **PATCH /api/admin/users/:id**  
  Обновить роль (`role`) или статус блокировки (`is_blocked`) пользователя.  
  Ограничения: нельзя изменить собственный аккаунт; `admin` не может управлять `admin`/`super-admin`; только `super-admin` может назначать роль `super-admin`.

- **GET /api/admin/activity?limit=50&offset=0**  
  Журнал аудита с пагинацией. Возвращает `{ items: [...], total: N }`. Каждая запись включает действие, пользователя, IP, уровень.

- **GET /api/admin/deploys**  
  История деплоев из `/var/log/tasktime-deploy.log`. Возвращает массив объектов `{ id, started_at, status, summary }`. Если лог недоступен — пустой массив.

## Развитие API

Для обратной совместимости при добавлении новых полей или эндпоинтов рекомендуется версионирование путей (например `/api/v1/...`) при необходимости. Текущие эндпоинты без префикса версии считаются стабильным контрактом для интеграций.
