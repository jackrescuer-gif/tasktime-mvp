## 2. Доменная модель (целевой MVP)

- **Organization** (опционально) — Тенант; содержит Users, Teams, Projects.
- **User** — Учётная запись, роль; исполнитель/автор по задачам; участник команд.
- **Team** — Продуктовая команда; участники (user + роль); может быть привязана к проектам.
- **Project** — Контейнер задач, одна доска по умолчанию, бэклог, спринты.
- **Issue** — Единица работы; project_id, type, status, assignee, опционально sprint_id, parent_id (иерархия).
- **Comment** — Комментарий к задаче; автор, текст.
- **Sprint** — Временной бокс; project_id; у задач поле sprint_id.
- **Board** — У проекта есть доска(и); колонки (status_key, порядок, wip_limit).
- **BoardColumn** — Колонка доски; соответствует статусу; задачи группируются по статусу.
- **Label** — Метка в рамках проекта; связь многие-ко-многим с задачами.
- **TimeLog** — Связь с задачей; пользователь, started_at, ended_at, длительность.
- **AuditLog** — Только добавление; действие, entity_type, entity_id, details.

Подробнее: **MVP_DOMAIN_MODEL.md** (ER-диаграмма и связи).

## 3. Расположение модулей (целевое)

| Модуль   | Ответственность                      | Примеры API                              |
|----------|--------------------------------------|------------------------------------------|
| auth     | Вход, регистрация, JWT, имперсонация | POST /api/auth/login, /register, GET /me |
| users    | Список пользователей, CRUD в админке | GET /api/users, PATCH /api/admin/users/:id |
| projects | CRUD проектов                       | GET/POST /api/projects, GET /api/projects/:id |
| issues   | CRUD задач, статус, переупорядочивание | GET/POST/PUT/PATCH/DELETE /api/issues/* |
| comments | Комментарии к задачам                | GET/POST /api/issues/:id/comments        |
| boards   | Доска и колонки                      | GET /api/projects/:id/board              |
| sprints  | CRUD спринтов, назначение задач      | GET/POST /api/projects/:id/sprints       |
| time     | Старт/стоп таймера, список           | POST /api/issues/:id/time/start\|stop    |
| teams    | Команды и участники                  | GET/POST /api/teams, …/members           |
| audit    | Чтение активности (админ)           | GET /api/admin/activity                  |
| admin    | Статистика, пользователи, деплои     | GET /api/admin/stats, /users, /deploys   |

В каждом модуле: **api.js** (маршруты) → **service.js** (логика) → **repository.js** (SQL). Импорт репозиториев между модулями запрещён.

См. **REPO_RESTRUCTURE_PLAN.md** (целевая структура папок и шаги переноса кода).

## 4. Дорожная карта миграции

| Фаза | Содержание | Риск   | Результат |
|------|------------|--------|-----------|
| **0** | Общий auth, errors, audit, config | Низкий | Подготовка к модулям |
| **1** | Выделение модулей (auth, users, projects, tasks, time, teams, admin) | Низкий | Тонкий server.js; модули по фичам |
| **2** | Новый домен: комментарии, спринты, доски, метки | Средний | Новые таблицы и API; без удаления старого |
| **3** | Объединение tasks + task_items → issues | Высокий | Единая модель Issue; скрипт миграции |
| **4** | Структура фронтенда; опционально организации | Низкий–средний | Поддерживаемый UI; опционально мультитенантность |

**Ограничение:** Деплой только с явного подтверждения пользователя. Каждая фаза — отдельно развёртываемая и тестируемая.

## 6. Принципы (кратко)

- **Простота важнее гибкости** — сначала MVP, расширение потом.
- **Скорость разработки** — Понятные модули и правила; удобно для AI.
- **Модульный монолит** — Один артефакт развёртывания; чёткие границы.
- **Безопасная миграция** — Пошагово; без «большого взрыва»; обратная совместимость до явного перехода.
- **Без переусложнения** — Без микросервисов и тяжёлых фреймворков в MVP.

Этот документ — отправная точка для решений по архитектуре и планированию реализации.
