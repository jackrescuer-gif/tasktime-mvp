# Production To Dev Sync And My Time Design

## Goal

Сделать так, чтобы:

- `production` был единственным источником истины для реального проекта `TTMP`, его спринтов, задач и списаний времени;
- `dev` можно было регулярно приводить к состоянию `production`, чтобы пайп разработки и проверки был максимально `1:1`;
- `seed` перестал быть источником "боевой" правды и использовался только для локального пустого старта;
- `My Time` показывал отдельно человеческое время и время/стоимость взаимодействия с ИИ, но при этом оставался журналом фактических списаний.

## Decision

Используем вариант `2`: отдельные БД для `production` и `dev`, где `production` является источником истины, а `dev` получает управляемую синхронизацию данных из `production`.

Это решение выбрано потому, что:

- общий PostgreSQL для `dev` и `production` слишком опасен и ломает изоляцию окружений;
- обычный `seed` не решает задачу реалистичного `1:1`, потому что dev и production быстро расходятся;
- направленный sync сохраняет предсказуемость, позволяет тестировать на копии реальных данных и не требует смешивать demo-данные с рабочими.

## Separation Of Concerns

Нужно жёстко разделить три разных сценария данных.

### 1. Bootstrap

`backend/src/prisma/bootstrap.ts` остаётся источником только для системных пользователей и минимального технического старта окружения:

- дефолтные пользователи;
- пароль через `BOOTSTRAP_DEFAULT_PASSWORD`;
- без проектов `TTMP`, спринтов, задач и логов времени.

### 2. Seed

`backend/src/prisma/seed.ts` остаётся только для локального/demo наполнения:

- локальная разработка с пустой БД;
- smoke/UAT на изолированном стенде;
- ручной старт окружения без production snapshot.

Но `seed` больше не считается обязательным источником данных для `dev` и тем более для `production`.

### 3. Sync

Реальные данные `TTMP` и связанные сущности должны попадать в `dev` через отдельный сценарий синхронизации:

- направление строго `production -> dev`;
- без обратной записи из `dev` в `production`;
- c журналом изменений и безопасным dry-run режимом.

## Data Sync Model

### Source Of Truth

В `production` живут реальные:

- `Project` (`TTMP` и связанные проекты при необходимости);
- `Sprint`;
- `Issue`;
- `Comment`;
- `TimeLog`;
- `AiSession`.

Именно эти сущности должны составлять sync-снимок для `dev`.

### Sync Semantics

Синхронизация должна быть идемпотентной:

- существующие записи в `dev` обновляются по стабильным ключам;
- отсутствующие записи создаются;
- записи, удалённые или архивированные в `production`, должны либо удаляться, либо явно помечаться как неактуальные в `dev` по выбранной стратегии;
- повторный запуск не должен плодить дубли.

Предпочтительная стратегия идентификации:

- `Project` по `key`;
- `Sprint` по составному ключу `project + name` либо по отдельному внешнему sync key;
- `Issue` по `project key + number`;
- `Comment`, `TimeLog`, `AiSession` через явные внешние sync-идентификаторы или детерминированные natural keys, если их достаточно.

Для долгосрочной устойчивости лучше добавить явные `externalSource` / `externalId` поля для сущностей, которые синхронизируются между БД. Это особенно важно для `Comment`, `TimeLog` и `AiSession`, где natural key ненадёжен.

### Execution Model

Нужен отдельный sync-скрипт или job, который:

1. Подключается к `production` в режиме чтения.
2. Выгружает допустимый набор сущностей `TTMP`.
3. Применяет этот snapshot в `dev`.
4. Печатает отчёт:
   - сколько объектов создано;
   - сколько обновлено;
   - сколько пропущено;
   - сколько удалено/деактивировано.

Сценарий запуска:

- вручную по команде;
- опционально по расписанию;
- опционально перед UAT на `dev`.

Автоматический sync на каждый deploy делать не стоит: это создаёт лишнюю связанность между кодовым релизом и операцией переноса данных.

## My Time Model

`My Time` должен опираться на фактические данные, а не на искусственные заполнители.

### Core Principle

Главный список `My Time` остаётся журналом реальных списаний:

- `HUMAN` — ручной ввод и таймер;
- `AGENT` — записи, созданные на основе `AiSession` и распределённые по задачам.

### Required Aggregates

На странице нужны отдельные агрегаты:

- `Human spent`;
- `AI spent`;
- `AI cost`;
- `Total spent = Human + AI`.

Эти значения уже естественно строятся из `TimeLog.source` и `costMoney`, но должны быть переосмыслены как официальная продуктовая модель, а не только demo-отображение.

### AI Interaction Accounting

Гибридная модель означает:

- человеческое время не смешивается с AI-временем в одной неразличимой сумме;
- взаимодействие с ИИ учитывается отдельно, но входит в общую картину по задаче и пользователю;
- источник `AiSession` остаётся первичным носителем деталей сессии: модель, провайдер, токены, стоимость, длительность;
- `TimeLog(source = AGENT)` остаётся нормализованным слоем для UI и отчётов.

### Unallocated AI Time

Если AI-сессия не была разнесена по задачам, есть два допустимых режима:

1. запрещать создание такой сессии на уровне API;
2. разрешать, но показывать её как `Unallocated AI time`.

Для MVP лучше выбрать первый вариант: каждая `AiSession` должна либо иметь `issueId`, либо содержать `issueSplits`, из которых можно создать `AGENT`-логи.

## Security And Operational Rules

### Production Safety

- `production deploy` не должен запускать demo `seed`;
- bootstrap пользователей может оставаться отдельной одноразовой операцией;
- sync в `production` работает только на чтение.

### Dev Safety

- импорт в `dev` может полностью перезаписывать синхронизируемую часть данных;
- локальные dev-правки в синхронизируемых сущностях считаются временными;
- sync должен иметь `dry-run`, чтобы сначала увидеть diff.

### Access And Secrets

- для sync нужен отдельный read-only доступ к production БД;
- секреты production и dev хранятся раздельно;
- команда sync не должна требовать write-доступа к production.

## Minimal Architecture Changes

Для реализации понадобятся изменения в трёх зонах.

### 1. Backend / Prisma

- определить sync-стратегию и внешние ключи для синхронизируемых сущностей;
- возможно расширить Prisma schema полями синхронизации;
- вынести demo-данные из критического production-пути;
- добавить отдельный script/service для snapshot import/export.

### 2. Deploy / Ops

- не запускать `seed` в production flow;
- добавить отдельную операцию `prod -> dev sync`;
- описать её в deploy/runbook документации.

### 3. Frontend / My Time

- сделать `My Time` явным представлением гибридного учёта;
- при необходимости добавить отдельные summary-блоки или breakdown по задачам;
- корректно обрабатывать AI-сессии и связанные `AGENT`-логи как часть рабочего журнала, а не как тестовую особенность.

## Acceptance Criteria

Дизайн считается реализованным корректно, когда:

- реальный проект `TTMP` и связанные данные создаются и живут в `production`, а не зависят от demo-seed;
- `dev` можно синхронизировать с `production` без дублей и ручной правки БД;
- синхронизация однонаправленная и безопасная для `production`;
- `My Time` показывает отдельно `Human`, `AI`, `AI cost` и общий итог;
- AI-время отображается только на основе фактических `AiSession` / `AGENT`-логов;
- production deploy и sync описаны как разные операции.

## Risks

- без явных внешних sync-ключей синхронизация дочерних сущностей будет хрупкой;
- если включить sync прямо в deploy, отладка релизов станет сложнее;
- если оставить `seed` источником части production-данных, быстро появится двойная правда;
- если разрешить AI-сессии без привязки к задачам, `My Time` станет непредсказуемым.

## Recommendation

Идти в два шага:

1. Сначала сделать безопасный `prod -> dev` sync для `TTMP`, `Sprint`, `Issue`, `TimeLog`, `AiSession`.
2. Затем поверх этого дооформить `My Time` как официальный гибридный отчёт по `HUMAN` и `AGENT`.
