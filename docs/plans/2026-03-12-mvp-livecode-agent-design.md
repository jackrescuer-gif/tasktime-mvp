## MVP LiveCode — задачи для агента и человека

### Цель

Сделать так, чтобы:
- задачи по разработке Flow Universe MVP (проект **MVP LiveCode**) жили как обычные `Issue` в системе;
- у каждой задачи можно было указать, кто её выполняет — **агент** или **человек**;
- из Cursor можно было запросом вроде «Покажи все задачи, активные для реализации» получить срез этих задач;
- задачи, помеченные как агентские, автоматически попадали в очередь задач ИИ (GSD/Codex) и синхронизировались по статусам.

---

### Доменная модель и БД

1. **Проект `MVP LiveCode`**
   - Реализуется как обычный `Project` с уникальным `key`, например `LIVE`.
   - Все задачи по самому продукту (мета‑уровень) заводятся как `Issue` внутри этого проекта.

2. **Расширение `Issue` для ИИ**
   - Новые поля:
     - `aiEligible: Boolean @default(false)` — задача может быть выполнена агентом (чекбокс в UI).
     - `aiExecutionStatus: AiExecutionStatus @default(NOT_STARTED)` — состояние именно выполнения агентом:
       - `NOT_STARTED | IN_PROGRESS | DONE | FAILED`.
     - (опционально) `aiAssigneeType: AiAssigneeType @default(HUMAN)` — кто считается основным исполнителем:
       - `HUMAN | AGENT | MIXED`.
   - Индексы:
     - по `projectId`, `status`, `aiEligible` для быстрого выборочного отображения активных задач для реализации.

3. **Связь с `AiSession` и учётом времени**
   - Уже есть `AiSession` и `TimeLog` с полем `source = HUMAN | AGENT`.
   - Для задач, которые делает агент, можно:
     - логировать время агента через `TimeLog` c `source = AGENT`, связанное с `AiSession`;
     - при закрытии GSD‑таска обновлять `aiExecutionStatus` и, при необходимости, статус `Issue`.

---

### API и фильтры

1. **Получение активных задач для реализации**
   - Новый endpoint, например `GET /api/mvp-livecode/issues/active` **или** расширение существующего списка задач с фильтром:
     - критерии по умолчанию:
       - `project.key = 'LIVE'` (или id проекта MVP LiveCode),
       - `status IN (OPEN, IN_PROGRESS, REVIEW)`,
       - опционально — исключаем `CANCELLED` и `DONE`.
     - параметры фильтрации:
       - `onlyAiEligible=true|false` — вернуть только задачи, помеченные как агентские;
       - `assigneeType=AGENT|HUMAN|ALL` — фильтрация по типу исполнителя.

2. **Обновление агентского флага**
   - Endpoint для изменения `aiEligible` и, при необходимости, `aiAssigneeType`:
     - `PATCH /api/issues/:id/ai-flags`
     - тело: `{ aiEligible?: boolean, aiAssigneeType?: 'HUMAN' | 'AGENT' | 'MIXED' }`
   - RBAC:
     - менять флаг могут как минимум `MANAGER` и `ADMIN`,
     - опционально — сам `assignee` задачи (ограничение обсуждаемо).

3. **Синхронизация со статусами агента**
   - Endpoint для обновления `aiExecutionStatus`:
     - `PATCH /api/issues/:id/ai-status`
     - тело: `{ aiExecutionStatus: 'NOT_STARTED' | 'IN_PROGRESS' | 'DONE' | 'FAILED' }`
   - Используется:
     - интеграцией с GSD/Codex при изменении состояния todo,
     - либо самим агентом в Cursor, когда он завершил задачу.

---

### UI (веб‑приложение Flow Universe)

1. **Проект `MVP LiveCode`**
   - В списке проектов отображается как обычный проект с ключом `LIVE`.
   - Страница проекта:
     - стандартная доска/таблица задач;
     - преднастроенное представление **«Active for implementation»**.

2. **Чекбокс “делает агент”**
   - В таблице/борде задач:
     - отдельная колонка/значок **Agent** с чекбоксом;
     - при включении/выключении:
       - вызывается `PATCH /api/issues/:id/ai-flags`;
       - UI показывает индикатор, что задача видна/не видна агенту.
   - В карточке задачи:
     - блок «Исполнитель»:
       - `Тип исполнителя: Человек / Агент / Совместно`;
       - чекбокс **«Задачу может выполнить агент»**;
       - отображение `aiExecutionStatus` (например, "Агент: в работе / выполнено / ошибка").

3. **Представление “Активные для реализации”**
   - Фильтры:
     - статус ∈ {OPEN, IN_PROGRESS, REVIEW};
     - опционально — только текущий активный спринт проекта LIVE;
     - опциональное поле «Исполнитель: Все / Агент / Человек».
   - Это представление служит визуальным аналогом команды в Cursor «Покажи все задачи, активные для реализации».

---

### Интеграция с агентом (GSD / Codex / Cursor)

1. **Отображение в Cursor**
   - Skill / команда в Codex/Cursor:
     - при фразе вида «Покажи все задачи, активные для реализации»:
       - вызывает `GET /api/mvp-livecode/issues/active`;
       - показывает список задач с:
         - `KEY`, `title`, `status`, `priority`,
         - `aiEligible`, `aiExecutionStatus`, `aiAssigneeType`.

2. **Синхронизация с GSD‑todo**
   - При `aiEligible = true`:
     - создаётся/обновляется GSD‑todo c:
       - заголовком `LIVE-XX - Summary`,
       - описанием, ссылкой на web‑карточку, приоритетом;
       - метаданными: id issue, тип исполнителя, статус.
   - При `aiEligible = false`:
     - GSD‑todo архивируется/помечается как human‑only и исключается из списка для агента.

3. **Обновление состояния из мира агента**
   - При смене состояния GSD‑todo:
     - интеграция вызывает `PATCH /api/issues/:id/ai-status`;
     - опционально:
       - если `aiExecutionStatus = DONE`, можно:
         - либо переводить `Issue.status` в `REVIEW` / `DONE` (подтверждение человеком),
         - либо добавлять комментарий от агента с резюме выполненной работы.

---

### Безопасность и аудит

- Все изменения `aiEligible`, `aiAssigneeType`, `aiExecutionStatus` попадают в `AuditLog`:
  - `entityType = "Issue"`, `entityId = issue.id`;
  - `action` вроде `"issue.ai.flags.updated"` и `"issue.ai.status.updated"`;
  - `details` с предыдущими/новыми значениями и источником (`USER` или `AGENT_INTEGRATION`).
- RBAC:
  - чтение полей ИИ — доступно всем, кто видит задачу;
  - изменение флагов и статусов — минимум `MANAGER`/`ADMIN` или владелец задачи по настройке.

---

### НФ‑ограничения и эволюция

- Запрос «Покажи все задачи, активные для реализации» должен укладываться в общие NFR API (< 200 мс p95, разумные лимиты и пагинация).
- MVP:
  - один специальный проект `MVP LiveCode`;
  - чекбокс и фильтры только в рамках этого проекта.
- В будущем:
  - возможность помечать как агентские задачи в любых проектах;
  - более гибкая маршрутизация по ролям/типам агентов;
  - отдельный UI‑интерфейс очереди задач для всей ИИ‑команды (см. PIXEL_AGENTS_AND_TASKS).
