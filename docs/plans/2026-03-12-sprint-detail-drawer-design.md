# Sprint Detail Drawer Design

## Goal

Добавить на страницы `projects/:id/sprints` и `sprints` возможность открыть детализацию выбранного спринта, увидеть список его задач с ключевыми атрибутами и перейти в карточку каждой задачи.

## Decision

Используем общий `Drawer` для drill-down по спринту вместо отдельной страницы. Это сохраняет контекст списка спринтов, уменьшает число переходов и позволяет переиспользовать один UI-компонент на проектной и глобальной страницах.

## Backend

- Добавить endpoint `GET /sprints/:id/issues`.
- Ответ должен включать:
  - краткую информацию о спринте (`id`, `name`, `state`, `goal`, `project`, команды, даты, `_count`, `stats`);
  - массив `issues`.
- Для каждой задачи вернуть поля:
  - `id`
  - `projectId`
  - `number`
  - `title`
  - `type`
  - `status`
  - `priority`
  - `updatedAt`
  - `assignee { id, name }`
  - `project { id, key, name }`

## Frontend

- Добавить в `frontend/src/api/sprints.ts` клиент для загрузки задач спринта.
- Добавить новый тип ответа `SprintDetailsResponse` в `frontend/src/types/index.ts`.
- Создать общий компонент `SprintIssuesDrawer`.
- На `SprintsPage`:
  - оставить текущий список спринтов и backlog;
  - добавить действие открытия детализации выбранного спринта;
  - внутри drawer показать таблицу задач спринта.
- На `GlobalSprintsPage`:
  - добавить действие открытия детализации на карточке спринта;
  - использовать тот же drawer.

## UX

- Drawer открывается без смены маршрута.
- Drawer должен визуально наследовать текущую dark Linear-like тему приложения, а не выглядеть как стандартный Ant Design overlay.
- В шапке drawer отображаются:
  - название спринта;
  - статус;
  - проект;
  - даты;
  - goal, если есть;
  - количество задач.
- Таблица задач показывает:
  - ключ задачи (`PROJECT-123`);
  - название;
  - тип;
  - статус;
  - приоритет;
  - исполнителя;
  - дату обновления.
- Клик по ключу или названию ведёт на `IssueDetailPage`.
- Для пустого спринта показывается `Empty`.

## Visual Polish

- Приоритет для первой итерации: сначала читаемость и адаптивность, затем умеренный visual polish без радикальной смены UX.
- Drawer использует:
  - затемнённый scrim;
  - собственный header/body skin в токенах `tt-*`;
  - summary-блок `Sprint overview`;
  - отдельный блок `Issues in sprint` с таблицей в стиле `tt-table`.
- Статусы, приоритеты и типы лучше отображать через project-specific pill/dot styles, а не через базовые разношёрстные `Tag color=...`, чтобы тема была консистентной.
- На узких экранах drawer должен уменьшаться по ширине без ломания layout; таблица должна скроллиться внутри себя по `x`, а не раздвигать весь overlay.

## Testing

- Backend: добавить API-тест на `GET /sprints/:id/issues`.
- Frontend: проверить вручную открытие drawer и переход в карточки задач на обеих страницах.
- После изменений прогнать релевантные тесты и линтеры по изменённым файлам.
