# Sprint Issue Preview Drawer Design

## Goal

Добавить на страницу проекта `projects/:id/sprints` второй вложенный `drawer` для быстрого просмотра задачи из деталей спринта без потери контекста страницы.

## Decision

Используем read-only `IssuePreviewDrawer`, который открывается поверх `SprintIssuesDrawer` по клику на задачу. Полноценное редактирование и все mutation-действия остаются только на отдельной странице задачи `issues/:id`.

## UX

- `SprintIssuesDrawer` остаётся главным drawer со списком задач спринта.
- Клик по названию задачи или по строке открывает `IssuePreviewDrawer`.
- Клик по коду задачи (`PROJECT-123`) ведёт на отдельную страницу задачи.
- В `IssuePreviewDrawer` есть явная кнопка `Редактировать`, которая тоже ведёт на `issues/:id`.
- Закрытие preview drawer не закрывает drawer спринта.
- Закрытие drawer спринта одновременно сбрасывает выбранную задачу и закрывает preview drawer.

## Preview Scope

В preview показываем только данные для чтения:

- ключ задачи;
- название;
- тип;
- статус;
- приоритет;
- проект;
- исполнитель;
- автор;
- дату создания;
- описание;
- родительскую задачу;
- список подзадач.

В preview не показываем interactive editing:

- смену статуса;
- запуск и остановку таймера;
- ручной лог времени;
- создание и удаление комментариев;
- переключение AI execution полей;
- любые другие мутации.

## Frontend Structure

- Создать новый компонент `frontend/src/components/issues/IssuePreviewDrawer.tsx`.
- Компонент сам загружает данные задачи через существующий API-клиент `issuesApi.getIssue`.
- `SprintIssuesDrawer` хранит `selectedIssueId` и управляет открытием preview drawer.
- Внутри таблицы задач:
  - ключ задачи остаётся ссылкой на `issues/:id`;
  - название задачи становится кнопкой или ссылкой, которая открывает preview drawer без смены маршрута.

## Reuse Strategy

Не встраиваем `IssueDetailPage` целиком внутрь drawer. Текущая страница содержит edit-логику, формы, timer controls и другую функциональность, которая не нужна в режиме быстрого просмотра. Preview drawer собирается как отдельный лёгкий read-only компонент на тех же API-данных.

## Visual Rules

- Preview drawer визуально должен быть согласован с текущей `tt-*` dark theme.
- Используем такую же структуру панели, как на странице задачи: header, основной контент и компактный aside с ключевыми полями.
- На узких экранах drawer должен уменьшаться по ширине и не ломать layout основного sprint drawer.

## Testing

- Проверить вручную открытие preview drawer из sprint drawer.
- Проверить переход на `issues/:id` по коду задачи.
- Проверить переход на `issues/:id` по кнопке `Редактировать`.
- Проверить, что в preview нет controls для изменения данных.
- Прогнать релевантные frontend-проверки по изменённым файлам.
