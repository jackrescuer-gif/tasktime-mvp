# Что было сделано при запуске TaskTime (для отчёта / объяснения)

Краткое описание шагов, чтобы можно было понятно рассказать, что сделано, в том числе людям без технического бэкграунда.

## 1. Подготовка сервера

- **Где:** виртуальная машина (VPS) в облаке Timeweb, операционная система Ubuntu.
- **Что сделано:** на сервер установлены и настроены:
  - **Node.js** — среда для работы приложения;
  - **PostgreSQL** — база данных (задачи, пользователи, учёт времени);
  - **Nginx** — веб-сервер, который принимает запросы из интернета и передаёт их приложению;
  - **системный сервис** — чтобы приложение автоматически запускалось и перезапускалось при сбоях;
  - **файрвол (UFW)** — ограничен доступ к серверу только по нужным портам (сайт, SSH);
  - **Certbot** — для возможности в будущем включить HTTPS (защищённое соединение).

Итог: сервер готов к размещению приложения и базе данных.

## 7. Автоматический деплой через GitHub Actions

- **Когда сделано:** март 2026 г.
- **Что такое GitHub Actions:** встроенный в GitHub сервис автоматизации. Можно настроить, чтобы при определённом событии (например, публикации нового кода) автоматически выполнялась цепочка действий на сервере.
- **Что сделано:**
  - Настроен автоматический деплой: как только разработчик публикует изменения в GitHub (`git push`), сервер сам забирает новый код и перезапускает приложение — без ручного входа на сервер.
  - Сгенерирован специальный SSH-ключ только для деплоя (не от имени администратора — безопасно).
  - Ключ хранится в зашифрованных переменных GitHub (Secrets) — в коде проекта он нигде не виден.
  - На сервере разрешено перезапускать только сервис приложения без пароля — больше ничего лишнего.
  - Скрипт деплоя улучшен: ведёт журнал каждого обновления, пропускает деплой если новых изменений нет, и откатывается назад если новая версия не запустилась.
- **Результат:** деплой занимает ~15 секунд, полностью автоматический. Статус каждого деплоя виден в GitHub во вкладке Actions (зелёная галочка = успех, красный крест = проблема).

## 8. Исправление: авторизация не работала в Safari и Chrome после входа

- **Когда:** март 2026 г.
- **Проблема:** после успешного входа (логин + пароль) пользователь видел «пасхалку с котиком» вместо рабочего приложения — воспроизводилось в Chrome и Safari (в том числе на iPhone).
- **Причина:** сервер проверял наличие куки при загрузке страницы `/app`. Некоторые браузеры (особенно Safari с защитой ITP) не включают куку, установленную через JavaScript-запрос, в следующий навигационный запрос — в результате сервер считал, что пользователь не авторизован, и показывал страницу-заглушку.
- **Что исправлено:** сервер теперь отдаёт приложение без проверки куки. Проверка авторизации перенесена полностью на клиент (браузер): при загрузке приложение само проверяет наличие токена, при необходимости запрашивает сервер — и только если авторизация не подтверждена, показывает заглушку.
- **Пасхалка сохранена:** она по-прежнему появляется, если кто-то пытается открыть `/app` напрямую без входа (например, скопировав ссылку без учётки). Так и должно быть — это намеренная защита.
- **Результат:** авторизованные пользователи попадают в приложение без ошибок во всех браузерах. Попытки открыть приложение без входа по-прежнему заблокированы с пасхалкой.

## Админ-панель TaskTime — Март 2026

**Что сделано:** Создана отдельная панель управления системой для администраторов, доступная по адресу `/admin`. Доступ только по ролям `admin` и `super-admin`.

**Зачем:** Ранее у администраторов не было единого места для мониторинга работоспособности системы, управления пользователями и просмотра истории деплоев. Данные были разбросаны по разным разделам или недоступны вовсе.

**Что добавлено:**
- Страница `/admin` → `frontend/admin.html` — самостоятельная SPA с проверкой роли на клиенте
- Новый API-маршрут `/api/admin/*` с `adminMiddleware` (проверка роли на уровне сервера)
- `GET /api/admin/stats` — показатели доступности: аптайм приложения и сервера, статус и задержка БД, RAM, load avg, ошибки за 24ч
- `GET /api/admin/users` — список пользователей с ролями, статусом блокировки, последней активностью
- `PATCH /api/admin/users/:id` — изменение роли и блокировка/разблокировка пользователя; `admin` не может управлять другими `admin`/`super-admin`
- `GET /api/admin/activity` — журнал аудита с пагинацией и фильтрацией
- `GET /api/admin/deploys` — история деплоев из `/var/log/tasktime-deploy.log`
- Новая роль `super-admin` — расширенные права над admin-учётками
- Колонка `is_blocked` в таблице `users` (graceful migration — `ADD COLUMN IF NOT EXISTS`)
- Проверка блокировки при входе в систему
- Ссылка «Админ-панель» в боковой панели `app.html` для ролей `admin` и `super-admin`

**Результат:** Администраторы видят в одном месте статус системы в реальном времени, могут управлять учётками пользователей (в т.ч. блокировать) и просматривать историю деплоев. Простые пользователи получают 403 при попытке зайти на `/admin`.

**Как применить на сервере:**
```bash
# Применить миграцию is_blocked (выполнить один раз)
sudo -u tasktime psql -d tasktime -c "ALTER TABLE users ADD COLUMN IF NOT EXISTS is_blocked BOOLEAN NOT NULL DEFAULT FALSE;"
# Затем деплой как обычно
```

## Март 2025 — Исправление аутентификации в админ-панели

**Что было сделано:** Устранена цепочка ошибок, из-за которых пользователи не могли попасть в администраторскую панель — их перебрасывало обратно на страницу входа.

**Зачем:** Несмотря на то что код панели был написан, войти в неё не удавалось из-за трёх взаимосвязанных проблем.

**Что исправили:**
1. **Корневая причина — HTTP без TLS.** Сайт работает по протоколу HTTP. Сессионная кука была выставлена с флагом «только для защищённых соединений» (Secure), поэтому браузер отказывался отправлять её по HTTP. Страница входа в панель теперь проверяет сессию через токен (тот же механизм, что использует основное приложение), а не через куку.
2. **Неправильный порядок загрузки.** Данные начинали грузиться до того как страница успевала показаться, и при ошибке загрузки пользователь выбрасывался обратно. Теперь страница сначала показывается, а данные подгружаются фоном.
3. **Умный повтор запроса.** Если основной способ авторизации не срабатывает, система автоматически пробует запасной вариант, прежде чем отправить пользователя на страницу входа.

**Результат:** Администраторы могут заходить в панель управления, видеть статус системы, список пользователей, журнал активности и историю деплоев.

**Итог одной фразой:**  
В TaskTime появились удобные кнопки, которые позволяют за пару кликов превратить входящую задачу в сторю на Kanban-доске реального проекта и сохранить прозрачную связь между ними.

## Март 2026 — Модульный монолит бэкенда (рефакторинг структуры)

**Что сделано:**  
Бэкенд перестроен в модульную структуру без изменения поведения и схемы БД. Вся логика вынесена из одного файла `server.js` в модули по доменам (auth, users, projects, issues, time, admin) и общие компоненты (shared: auth, audit, errors).

**Зачем:**  
В одном большом `server.js` было сложно ориентироваться людям и ИИ-инструментам: маршруты, бизнес-логика и SQL смешаны в одном месте. Модульный монолит упрощает навигацию и поддержку: каждый модуль содержит только маршруты (api), бизнес-логику (service) и доступ к БД (repository).

**Что изменилось:**  
- `server.js` теперь только инициализирует Express, подключает middleware, монтирует роутеры модулей и отдаёт статику/страницы входа и приложения.  
- Созданы каталоги `backend/shared/` (auth, audit, errors) и `backend/modules/` с подкаталогами auth, users, projects, issues, time, admin. В каждом модуле: `api.js` (маршруты), `service.js` (логика), `repository.js` (SQL).  
- Конфигурация вынесена в `backend/config.js`. Аудит перенесён в `shared/audit.js`, старый `backend/audit.js` удалён.

**Результат:**  
Поведение API и приложения для пользователя не изменилось. Разработчикам и ИИ проще находить нужный код и вносить изменения: маршруты — в api.js, правила доступа и валидация — в service.js, запросы к БД — только в repository.js. Подробное описание — в `docs/refactor/backend_modularization.md`.
