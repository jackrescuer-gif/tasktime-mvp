# Документация TaskTime

## Назначение файлов

| Файл | Для кого | Содержание |
|------|----------|------------|
| **DEPLOY.md** | Тот, кто деплоит | Команды, порядок шагов, ссылка на репо, проверка. |
| **DEPLOYMENT_STEPS.md** | Нетехнари, отчёт | Что сделано простыми словами (сервер, БД, код, доступ, учётки). Одна фраза-итог в конце. |
| **ACCOUNTS.md** | Все | Формат входа (логин = email, пароль). Таблицы: демо-пользователи, учётки команды. Роли (admin, manager, user). |

## Стиль для DEPLOYMENT_STEPS

- Короткие абзацы, без жаргона где возможно.
- Разделы: подготовка сервера, база данных, размещение кода, доступ из интернета, учётки, обновления.
- В конце — одна итоговая фраза для отчёта.

## Стиль для ACCOUNTS

- Явно: логин = email (латиница), пароль отдельно.
- Таблицы: имя (латиница), логин (email), пароль, роль.
- Указать, что учётки создаются скриптом (`backend/scripts/seed.js`, на сервере — `init-db.sh`).

## Синхронизация

- При изменении списка пользователей в `backend/scripts/seed.js` обновить ACCOUNTS.md (логины, пароли, роли).
- При изменении шагов деплоя обновить DEPLOY.md и при необходимости DEPLOYMENT_STEPS.md.

## Правило: обновлять документацию после каждой задачи

В проекте действует cursor rule (`.cursor/rules/docs-after-every-task.mdc`): **документация — часть задачи, не отдельный шаг**.

### Что обязательно обновлять

| Документ | Когда |
|----------|-------|
| `DEPLOYMENT_STEPS.md` | После любой технической задачи — новый раздел: что сделано, зачем, результат, когда |
| `DEPLOY.md` | Изменился процесс деплоя, команды, инфраструктура |
| `docs/ENG/ADMIN_GUIDE.md` или `docs/RU/ADMIN_GUIDE.md` | Изменились процедуры администрирования |
| `docs/ENG/API.md` или `docs/RU/API.md` | Добавился или изменился API |
| `ACCOUNTS.md` | Изменились роли или учётки |
| `backend/README.md` | Изменилась внутренняя архитектура бэкенда |

### Стиль записи в DEPLOYMENT_STEPS.md

Писать так, чтобы ИТ-менеджер без технического бэкграунда понял и мог объяснить другим:
- **Что сделано** — простым языком
- **Зачем** — какую проблему решили
- **Результат** — что изменилось для пользователя/команды
- **Когда** — месяц и год

### Порядок работы

1. Выполнить задачу
2. Обновить `DEPLOYMENT_STEPS.md` (и другие документы при необходимости)
3. Залить всё **одним коммитом** (`git add`, `commit`, `push`)
