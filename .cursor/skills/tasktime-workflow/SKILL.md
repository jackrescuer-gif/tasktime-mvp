# Workflow TaskTime: полный пайплайн разработки

## Обязательная последовательность

Каждая задача на разработку проходит **все этапы по порядку**. Пропускать этапы — нельзя.

```
[1] GATEKEEPER         → архитектурное заключение (MCP validate_and_improve)
       ↓
[2] СИСТЕМНАЯ АНАЛИТИКА → требования, use-cases, API-контракт (system-analyst)
       ↓
[3] ИБ-СОГЛАСОВАНИЕ    → риски, auth, данные, куки, роли (infosec)
       ↓
[4] ТЕСТЫ              → unit, QA авто+ручные, UAT (tester)
       ↓
[5] РАЗРАБОТКА         → реализация по согласованному плану (developer)
       ↓
[6] ТЕСТИРОВАНИЕ       → прогон тестов, кросс-браузерный чек (tester)
       ↓
[7] ПРОВЕРКА С ПАВЛОМ  → показать результат, получить явное «ок»
       ↓
[8] ДЕПЛОЙ             → push в main → GitHub Actions (deploy-tasktime)
       ↓
[9] РАЗБОР НА ЗНАНИЯ   → git diff → извлечь паттерны/грабли → обновить скиллы агентов
```

---

## Этап 1 — Gatekeeper: архитектурное заключение

**Кто:** MCP `user-tasktime-gatekeeper`  
**Инструмент:** `validate_and_improve(request)`

Перед любым кодом вызвать gatekeeper:

```
validate_and_improve({ request: "<описание задачи>" })
```

Gatekeeper:
- проверяет соответствие архитектуре TaskTime
- выявляет риски и edge cases
- возвращает улучшенный промпт / рекомендации

Если задача затрагивает схему БД — дополнительно вызвать `check_schema(sql)`.  
Если нужен общий контекст стека — `get_context(topic)`.

**Результат этапа:** архитектурное заключение gatekeeper зафиксировано.

---

## Этап 2 — Системная аналитика

**Кто:** роль `system-analyst`  
**На основании:** заключения gatekeeper

Что делается:
- Формализовать требования (что система должна делать)
- Описать use-cases / сценарии использования
- Определить API-контракт (эндпоинты, форматы, коды ответов)
- Выявить граничные случаи и зависимости

**Результат этапа:** согласованная спецификация задачи.

---

## Этап 3 — Согласование с ИБ

**Кто:** роль `infosec`

Что проверяется:
- Разграничение доступа: какие роли видят/меняют что
- Аутентификация и сессии: нет ли cookie-gate антипаттерна, корректны ли флаги кук
- Передаваемые данные: что попадает в ответы API, нет ли утечки лишних полей
- Новые зависимости: нет ли known-уязвимостей

**Результат этапа:** ИБ-рекомендации учтены в спецификации.

---

## Этап 4 — Формирование тестов

**Кто:** роль `tester`

До написания кода описать:
- **Unit-тесты** — для каждой новой функции/модуля
- **API-тесты** — для каждого нового/изменённого эндпоинта
- **QA (авто)** — регрессионные сценарии
- **QA (ручные)** — чек-лист ручной проверки (включая кросс-браузерный, если затронут UI)
- **UAT** — приёмочные критерии: что пользователь должен увидеть/сделать

Если задача затрагивает auth/сессии — обязательно включить [Auth чек-лист](../tester/SKILL.md).

**Результат этапа:** тест-план согласован до старта разработки.

---

## Этап 5 — Разработка

**Кто:** роль `developer`

Реализация **строго по согласованной спецификации** (этап 2) с учётом ИБ-рекомендаций (этап 3).

Правила:
- Не отступать от API-контракта без нового согласования
- SPA-паттерн: HTML отдаём безусловно, auth только на API и клиенте
- Куки: `httpOnly`, `sameSite`, `secure: NODE_ENV==='production'`
- Не коммитить до прохождения тестов (этап 6)

---

## Этап 6 — Тестирование

**Кто:** роль `tester`

Прогнать все тесты из тест-плана (этап 4):
- Unit / API тесты — в CI или локально
- Ручной чек-лист (кросс-браузер для UI-задач)
- UAT — проверить приёмочные критерии

**Все тесты зелёные** → переход к этапу 7.  
Есть красные тесты → вернуться к этапу 5.

---

## Этап 7 — Проверка с Павлом

Показать результат пользователю:
- Демо или описание изменений
- Результаты тестирования
- Ожидаемое поведение

**Явное «ок» / «делаем» / «согласен»** → переход к деплою.  
Без явного подтверждения — не деплоить.

---

## Этап 8 — Деплой

**Кто:** роль `deploy-tasktime`

После явного подтверждения пользователя:
1. `git add` всех изменений
2. `git commit` (осмысленный message)
3. `git push origin main`
4. GitHub Actions запускает `deploy.sh` на сервере автоматически
5. Проверить лог деплоя: `tail -f /var/log/tasktime-deploy.log`

**Документация** — обновить до коммита (правило `docs-after-every-task`):  
DEPLOYMENT_STEPS.md + при необходимости DEPLOY.md, ACCOUNTS.md, docs/ENG/API.md или docs/RU/API.md.

---

## Этап 9 — Разбор изменений на знания: наполнение скиллов агентов

**Кто:** любой агент, завершивший задачу  
**Когда:** после деплоя (или после проверки с Павлом, если деплой ещё не был)  
**Обязательно:** да — знания, полученные в задаче, должны остаться в команде

### Алгоритм

1. **Собрать изменения задачи:**
   ```
   git log --oneline -10          # что сделано в последних коммитах
   git diff HEAD~N HEAD --stat    # какие файлы затронуты
   ```

2. **Для каждого изменения задать вопросы:**
   - Что нового узнали? (паттерн, антипаттерн, грабли, команда, конфиг)
   - Какой роли это полезно? (developer / infosec / tester / deploy-tasktime / …)
   - Это уже есть в соответствующем скилле?

3. **Обновить скиллы** — добавить только то, чего там ещё нет:
   - Новый паттерн или антипаттерн → в скилл роли
   - Новая команда / конфиг / путь на сервере → в `deploy-tasktime`
   - Новая ИБ-ловушка → в `infosec`
   - Новый сценарий тестирования → в `tester`
   - Изменение процесса документирования → в `docs-tasktime`

4. **Обновить `DEPLOYMENT_STEPS.md`** (правило `docs-after-every-task`): что сделано, зачем, результат.

5. **Залить скиллы и доки одним коммитом** вместе с кодом (или отдельным `docs(skills): ...`).

### Что попадает в скиллы

| Тип знания | Куда |
|---|---|
| Паттерн / антипаттерн кода | `developer` |
| Грабли с auth, куками, RBAC | `infosec` + `developer` |
| Кросс-браузерные/регрессионные сценарии | `tester` |
| Команды деплоя, пути, конфиги, sudoers | `deploy-tasktime` |
| CI/CD, GitHub Actions, SSH | `network-infrastructure` |
| Новые API-эндпоинты или изменения схемы | `docs-tasktime` + `system-analyst` |
| Новые роли или учётки | `docs-tasktime` (ACCOUNTS.md синхронизация) |
| Новый архитектурный паттерн | `corporate-architect` |

### Что НЕ попадает в скиллы

- Одноразовые хотфиксы без переиспользуемой ценности
- Детали конкретных данных (ID, пароли, токены)
- Описание того, что и так очевидно из кода

### Критерий готовности этапа

Скиллы обновлены так, что **другой агент**, прочитав их, не наступит на те же грабли и сможет выполнить аналогичную задачу без дополнительных разъяснений.

---

## Исключения и мелкие правки

- **Опечатки, одна строка документации** — можно спросить «Исправить и залить?» и действовать по ответу; полный пайплайн не нужен.
- **Hotfix в prod** — gatekeeper + ИБ-проверка обязательны; аналитику и тесты можно провести ускоренно с явным указанием «hotfix».
- **Эксперимент / черновик** — разрабатывать можно, но деплоить только после полного пайплайна.
