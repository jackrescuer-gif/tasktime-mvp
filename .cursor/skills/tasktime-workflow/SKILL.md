# Skill: tasktime-workflow

Цель: описать простой, реалистичный workflow разработки для MVP с одним основным разработчиком.

## 1. Жизненный цикл задачи

1. **Понимание задачи**
   - Прочитать описание (TaskTime/доки/чаты).
   - Уточнить требования по архитектуре в `SYSTEM_ARCHITECTURE.md`, `REBUILD_PLAN_V2.md` и текущим модулям.

2. **Дизайн (минимальный)**
   - Решить, какие модули backend/frontend затрагиваются.
   - При необходимости сделать короткий набросок API/DTO.
   - Проверить, не дублирует ли задача уже существующую функциональность.

3. **Реализация**
   - Сначала backend (router → service → Prisma).
   - Затем frontend (React‑страницы/компоненты, Zustand, AntD).
   - Обновить/добавить Zod‑схемы и типы DTO.

4. **Тестирование**
   - Прогнать все тесты локально.
   - Для существенных изменений — добавить/обновить Vitest + Supertest тесты.
   - Минимум: руками пройти основной пользовательский сценарий задачи.
   - Для **каждой новой фичи** в ответах ИИ-разработчика **обязателен раздел "UAT / Приёмочные тесты"** с пошаговым чек-листом для проверки на dev-стенде (контекст, предусловия, сценарии, ожидаемый результат).

5. **Подтверждение и фиксация**
   - После реализации и локального тестирования описать, что сделано, и приложить UAT-чек-лист.
   - Дождаться явного **ОК от Павла по результатам проверки на dev-стенде**.
   - До получения ОК **не выполнять `git commit`, `git push` и не обновлять/dev-стенд** для этой фичи.

5. **Code review (опционально)**
   - Можно использовать AI как reviewer (описать, что изменено и почему).
   - Важно: проверить контракты API, статусы, RBAC и влияние на существующие кейсы.

6. **Деплой**
   - Слить изменения в ветку `main` или `claude/...` (по принятому процессу).
   - Убедиться, что билд и миграции проходят.
   - Проверить ключевые сценарии на стенде после деплоя.

## 2. Отношение к gatekeeper/MCP

- Gatekeeper MCP и сложные многоступенчатые pipeline’ы **не являются обязательными** для MVP.
- Если gatekeeper настроен — можно использовать его как дополнительную проверку (lint + тесты + simple rules).
- Нельзя блокировать задачу только потому, что MCP не настроен/недоступен.

## 3. Принципы

- **Сначала простота.** Выбирать самый простой вариант, который решает задачу и вписывается в архитектуру.
- **Один источник правды.** Backend‑контракты и Prisma‑схема — основа; фронт не должен придумывать свои модели.
- **Маленькие шаги.** Разбивать крупные изменения на несколько MR/коммитов с рабочим состоянием.
