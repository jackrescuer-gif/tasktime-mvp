# Тестировщик (авто + ручные тесты)

## Зона ответственности

- **Автотесты**: unit, API, e2e; фреймворки и пайплайны (CI); поддержка и регрессия.
- **Ручное тестирование**: сценарии, чек-листы, приёмочные критерии; исследовательское тестирование при необходимости.
- Приоритизация по рискам; отчётность по дефектам и покрытию.

## Обязательный контекст

Учитывать контекст импортозамещения: [ru-compliance-context](../ru-compliance-context/SKILL.md). система не ЗоКИИ; при выборе инструментов тестирования (в т.ч. иностранных) — допущения и легальные варианты использования; не предполагать автоматически режим КИИ.

## Выход

Тест-кейсы, автотесты, отчёты о тестировании; при необходимости — допущения по используемым инструментам.

## Обязательный чек-лист: авторизация и сессия

Любую задачу, затрагивающую логин / выход / защищённые страницы, **обязательно** проверять по этому чек-листу:

### Кросс-браузерное тестирование логина

| # | Сценарий | Chrome | Safari (desktop) | Safari (iPhone) | Firefox |
|---|---|---|---|---|---|
| 1 | Логин → попадаем в приложение | ✓ | ✓ | ✓ | ✓ |
| 2 | Логин → перезагрузка `/app` → всё ещё в приложении | ✓ | ✓ | ✓ | ✓ |
| 3 | Логин → закрыть вкладку → открыть `/app` снова → приложение (не логин) | ✓ | ✓ | ✓ | ✓ |
| 4 | Прямой переход на `/app` без логина → пасхалка или логин | ✓ | ✓ | ✓ | ✓ |
| 5 | Выход → попытка вернуться по кнопке «Назад» → не попасть в приложение | ✓ | ✓ | ✓ | ✓ |

> **Почему важно тестировать мобильный Safari отдельно:** iOS Safari (ITP) может не отправлять куки, установленные через `fetch()`, в следующем навигационном запросе — это исторически сломало auth-флоу в TaskTime (март 2026). Теперь app.html защищён на клиенте, но регрессию важно проверять.

### Auth-сценарии для каждого релиза

- [ ] Все роли (`admin`, `manager`, `user`, `cio`, `viewer`) — логин без ошибок
- [ ] После логина нет пасхалки (`?blocked=1` в URL)
- [ ] При недействительном токене — редирект на логин, не 500
- [ ] Приватный режим браузера (Private/Incognito) — логин работает
- [ ] Логин с неверным паролем — корректная ошибка, не 500

### Тестирование пасхалки (security UX)

- [ ] Прямой GET на `/app` без куки/токена → пасхалка
- [ ] Сохранённый URL `/app` открыт в новой вкладке без сессии → пасхалка
- [ ] После выхода → переход по `/app` → пасхалка (не приложение)
