# Правила проекта для AI-участников

Этот документ задаёт правила, которым должен следовать код, генерируемый AI, чтобы соответствовать целевой архитектуре репозитория TaskTime.

---

## Архитектурные правила

- **Бэкенд — модульный монолит.** Вся серверная логика живёт в одном приложении, разбитом на модули.
- **Каждая фича реализуется модулем** внутри `backend/modules/`.
- **Структура модуля обязательна:** в каждом модуле должны быть файлы:
  - `api.js` — HTTP-слой
  - `service.js` — бизнес-логика
  - `repository.js` — работа с данными

Не создавать модули без этой тройки. Не выносить логику из одного слоя в другой (например, SQL в `api.js` или `service.js`).

---

## Правила кода

- **`api.js`** — только обработка HTTP: приём запросов, вызов сервиса, формирование ответа. Без бизнес-логики и без прямых запросов к БД.
- **`service.js`** — вся бизнес-логика: валидация, оркестрация, вызовы репозитория. Без SQL и без деталей HTTP.
- **`repository.js`** — все запросы к базе данных. Только здесь допустимы SQL и вызовы слоя доступа к данным.

Соблюдать разделение слоёв: API → Service → Repository.

---

## Правила работы с базой данных

- **SQL только в `repository.js`.** Ни в `api.js`, ни в `service.js` не должно быть строк SQL или прямых обращений к БД.
- **Весь доступ к БД — через `db.js`.** Репозитории используют общий модуль подключения/пула; не создавать отдельные подключения в модулях.

---

## Правила размера файлов

- **Избегать файлов длиннее 300 строк.** Если файл разрастается — выносить логику в отдельные функции или подмодули.
- **Предпочитать небольшие функции.** Одна функция — одна зона ответственности; длинные функции разбивать.

---

## Зависимости

- **Не добавлять новые npm-зависимости без необходимости.** Перед добавлением пакета проверить, можно ли обойтись встроенным API или уже установленными библиотеками.
- При добавлении зависимости — обосновать в коде или в описании коммита.

---

## Правила документации

При выполнении **значительного рефакторинга** AI обязан создать отчёт в каталоге:

```
docs/refactor/
```

### Требования к отчёту

- **Язык:** русский.
- **Содержание:**
  - **Что изменилось** — перечень изменений (модули, файлы, сценарии) без кода.
  - **Почему изменилось** — причина рефакторинга (техдолг, масштабируемость, читаемость и т.п.).
  - **Какой архитектурный эффект достигнут** — что улучшилось с точки зрения структуры и поддерживаемости.

- **Не включать** в отчёт сырой код (сниппеты, листинги). Описывать только логику и решения.
- **Аудитория:** отчёт должен быть понятен нетехническому руководителю проекта (менеджеру), а не только разработчикам.

Цель — оставлять след рефакторинга в виде короткого, понятного документа для тех, кто не читает код.
